## 1. Определение и архитектура

**Буферизированный ввод-вывод** (стандартная библиотека `stdio`) — это реализация потокового ввода-вывода, где в оперативной памяти средствами библиотеки создаётся промежуточный буфер для хранения данных, читаемых из файла/устройства или записываемых в них. Средствами операционной системы осуществляется синхронизация этого буфера с файлом или устройством [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

![[Pasted image 20260315111849.png|642]]

Функции семейства `fopen`, `fread`, `fwrite`, `scanf`, `printf` являются буферизированными. Для каждой такой функции создаётся два уровня буферов:

### Многоуровневая архитектура прохождения данных

text

Данные → [Буфер библиотеки (пространство пользователя, уровень stdio)] → [Буферы ядра (page cache, буфер грязных страниц)] → [Устройство хранения (ПЗУ/блочное устройство)]

Путь прохождения данных (и в обратную сторону для чтения):

1. **Первый уровень (ОЗУ)** — буфер библиотеки в пространстве пользователя
    
2. **Второй уровень (ОЗУ)** — буферы ядра (кэш страниц)
    
3. **Третий уровень (ПЗУ)** — физическая запись на жесткий диск
    

При чтении данные проходят этот же путь в обратном направлении.

## 2. Назначение и область применения

Буферизированный ввод-вывод создан для работы с **большими объёмами данных**, постоянно передающимися потоками:

- Потоковые данные
    
- Музыкальные файлы
    
- Видео файлы
    
- Сетевые пакеты
    

### Принцип эффективности

Вместо выполнения множества мелких записей отдельными функциями, буферизация накапливает блоки данных в библиотеке, откуда они единовременно попадают в буфер ядра и записываются на диск.

**Объём буфера** чаще всего равен размеру блока файловой системы (см. Виртуальная файловая система), что позволяет выполнить запись за одну операцию. Это делает буферизированный ввод-вывод значительно эффективнее прямого файлового ввода-вывода при работе с большими объёмами.

**Ограничение**: при записи малых объёмов данных буферизированный ввод-вывод проигрывает по производительности прямому файловому вводу-выводу.

Управление синхронизацией данных с буферами ядра берёт на себя библиотека [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

## 3. Режимы работы буферов

Библиотека stdio поддерживает два основных режима буферизации, которые можно принудительно установить через `setvbuf`:

### 3.1 Построчная буферизация (`_IOLBF`)

**Условия активации**: включается при работе со стандартными потоками ввода/вывода (stdin, stdout, stderr), связанными с терминалом [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

**Характеристики**:

- Размер буфера обычно соответствует размеру строки (часто 255 символов)
    
- Символ `'\n'` (переход на новую строку) вызывает сброс (синхронизацию) буфера
    

**Поведение функции `printf`**:

- При наличии `'\n'` в строке вывода — буфер сбрасывается (синхронизируется) немедленно
    
- При выводе без `'\n'` данные накапливаются в буфере библиотеки до достижения его предельного размера, после чего происходит автоматический сброс
    

**Принудительная синхронизация**:

- `fflush(stdout)` — сбрасывает буфер библиотеки в буфер грязных страниц ядра
    
- `sync()` — сбрасывает буфер грязных страниц ядра на физическое устройство
    

### 3.2 Поблочная (полная) буферизация (`_IOFBF`)

**Условия активации**: включается при работе с файлами [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

**Характеристики**:

- Размер буфера равен размеру блока файловой системы
    
- Буфер не синхронизируется до достижения размера блока
    
- Если блок не заполнен, данные синхронизируются при завершении программы
    

**Важная особенность**: поток файлового ввода-вывода сбросить нельзя. Если файл используется другой программой, удалить файл невозможно.

### 3.3 Критическое поведение при перенаправлении

При перенаправлении вывода с терминала в файл происходит смена режима буферизации:

> Когда вывод перенаправляется с экрана в файл, всё, что выводится на экран, начинает работать не с построчной, а с поблочной буферизацией. Символ `'\n'`, который работал как триггер сброса в построчной буферизации, теряет это значение — вывод не будет виден до заполнения буфера [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

**Практическое следствие**:

- При работе с терминалом: буферизация построчная — помогают и `'\n'`, и `fflush`
    
- При работе с файлом: буферизация поблочная — синхронизация только при заполнении блока или вызове `fflush`
    

## 4. API буферизированного ввода-вывода

### 4.1 Открытие/закрытие потоков

c

FILE* fopen(const char* path, const char* mode);

Работа с файловым указателем (FILE*), а не с файловым дескриптором.

c

FILE* fdopen(int fd, const char* mode);

Преобразует файловый дескриптор в файловый указатель.

c

int fileno(FILE* stream);

Преобразует файловый указатель обратно в дескриптор (для смешанной работы).

c

int fclose(FILE* stream);

Аналог `close()` для файловых указателей.

### 4.2 Чтение и запись

c

size_t fread(void* ptr, size_t size, size_t nmemb, FILE* stream);

Параметры: буфер, размер элемента чтения, количество элементов, файловый указатель.

c

size_t fwrite(const void* ptr, size_t size, size_t nmemb, FILE* stream);

Параметры: буфер, размер элемента записи, количество элементов, файловый указатель.

Функции `fread` и `fwrite` работают с буфером как с массивом элементов указанного размера, возвращая количество успешно прочитанных/записанных элементов [](https://icpc.student.cs.uwaterloo.ca/~acm00/cppreference/doc/html/en/c/io/fread.html)[](https://zzapomni.com/namestnikov-programmirovanie-na-yazyke-2008-16444/48)[](https://metanit.com/c/tutorial/7.9.php).

### 4.3 Управление буферизацией

c

int fflush(FILE* stream);

**Назначение**: принудительно синхронизирует буфер библиотеки с буфером грязных страниц ядра [](https://man.docs.euro-linux.com/EL%207/man-pages/fflush.3.en.html)[](https://manpages.ubuntu.com/manpages/focal/en/man3/fflush.3.html).

**Важное замечание**: `fflush` сбрасывает данные только в буферы ядра, но не гарантирует немедленной физической записи на диск. Например, при выводе строки через `printf` без `'\n'` можно вызвать `fflush(stdout)` для попадания данных в буфер грязных страниц, но для немедленного вывода на экран потребуется дополнительный вызов `sync()`.

**Особенности для файлового дескриптора 1** (поток вывода):

- `fflush` отправляет данные в буфер грязных страниц
    
- `sync` инициирует сброс буфера грязных страниц на устройство
    

c

int fpurge(FILE* stream);

**Предупреждение**: функция сброса потока ввода. Использование этой функции сигнализирует о необходимости пересмотреть архитектуру программы — если она требуется, значит, в программе что-то работает не так [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

### 4.4 Позиционирование в потоке

c

int fseek(FILE* stream, long offset, int whence);

Перемещает указатель позиции (каретку) в потоке.

**Рекомендация**: функцию лучше никогда не использовать. Если необходимо, применяйте обёртки:

c

long ftell(FILE* stream);

Возвращает текущую позицию в потоке.

c

void rewind(FILE* stream);

Перемещает указатель в начало потока.

**Важно**: прыжки в потоке буферизированного ввода-вывода — плохая идея, так как могут привести к потере данных или нарушению целостности [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

### 4.5 Настройка режима буферизации

c

int setvbuf(FILE *restrict stream, char *restrict buf, int type, size_t size);

Позволяет принудительно изменить логику работы буферизации для указанного потока, установив тип буферизации (построчная/поблочная/небуферизированная) [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

## 5. Буфер упреждающего чтения и управление памятью

### 5.1 Область действия политики упреждающего чтения

Политика буфера упреждающего чтения (read-ahead) распространяется на:

- Файловый ввод-вывод
    
- Память, хранящуюся в области MMAP (динамические библиотеки и большие блоки выделенной памяти)
    

MMAP работает с файловыми дескрипторами, поэтому политика буфера упреждающего чтения автоматически применяется и к этой области.

### 5.2 Большие выделения памяти

При выделении большого объёма памяти через `malloc` он оказывается в области MMAP в адресном пространстве процесса, выполнившего этот системный вызов.

### 5.3 Управление через posix_madvise

c

int posix_madvise(void* addr, size_t len, int advice);

Функция позволяет управлять политикой буфера упреждающего чтения для области памяти [](https://manpages.debian.org/bookworm/manpages-ru-dev/posix_madvise.3)[](https://man.archlinux.org/man/extra/man-pages-ru/posix_madvise.3.ru).

**Значения параметра `advice`** [](https://manpages.debian.org/bookworm/manpages-ru-dev/posix_madvise.3)[](https://man.archlinux.org/man/extra/man-pages-ru/posix_madvise.3.ru):

- `POSIX_MADV_NORMAL` — поведение по умолчанию
    
- `POSIX_MADV_SEQUENTIAL` — ожидается последовательный доступ (все страницы можно заранее прочитать)
    
- `POSIX_MADV_RANDOM` — ожидается случайный порядок доступа (упреждающее чтение менее эффективно)
    
- `POSIX_MADV_WILLNEED` — доступ ожидается в ближайшем будущем (упреждающее чтение полезно)
    
- `POSIX_MADV_DONTNEED` — доступ не ожидается в ближайшем будущем
    

### 5.4 NUMA-специфичный пример использования

Рассмотрим систему NUMA (Non-Uniform Memory Access) с двумя процессорами, каждый из которых имеет собственные плашки оперативной памяти. Существует интерфейс, позволяющий одному процессору обращаться к памяти другого, но через медленный интерконнект.

**Сценарий**:

1. Запускается программа, создающая большой объём памяти
    
2. Требуется, чтобы оба процессора обрабатывали эту память раздельно
    
3. По умолчанию память выделяется в ОЗУ первого процессора
    
4. Второй процессор вынужден обращаться к чужой памяти через медленный интерфейс
    

**Результат по умолчанию**: теоретически ожидаемое ускорение в 2 раза не достигается — производительность оказывается значительно ниже из-за медленных межпроцессорных обращений.

**Причина**: при работе с файлами (и областями MMAP) активируется буфер упреждающего чтения. Часть страниц выделяется, и упреждающе выделяется вся последующая порция, которая ложится в области памяти того процессора, который инициировал выделение.

**Решение через `posix_madvise`**:

Если с помощью `posix_madvise` указать флаг `POSIX_MADV_RANDOM` (случайный доступ к области), то:

- Вся память выделяться не будет
    
- Страницы будут выделяться постранично, по мере фактического обращения
    
- Когда первый процессор обращается к своей странице — она выделяется в его оперативной памяти
    
- Когда второй процессор обращается к своей странице — она выделяется в его памяти
    

**Итог**: сегмент памяти остаётся общим логически, но физически располагается на разных узлах NUMA, обеспечивая локальность доступа для каждого процессора и оптимальную производительность [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).

## 6. Ключевые особенности и предостережения

1. **Сброс буфера грязных страниц**: функция `sync` позволяет инициировать сброс буфера грязных страниц в момент, требуемый приложению, а не когда решит система.
    
2. **Смешивание потоков и дескрипторов**: при необходимости работы с файлом в двух режимах используется преобразование через `fileno` или `fdopen`.
    
3. **Незаполненный буфер**: при поблочной буферизации незаполненный буфер синхронизируется только при завершении программы.
    
4. **Блокировка файлов**: если файл используется другой программой, удалить его невозможно.
    
5. **Влияние на производительность в NUMA-системах**: политика упреждающего чтения может негативно влиять на распределение памяти в многопроцессорных системах с неоднородным доступом к памяти, что корректируется через `posix_madvise`.

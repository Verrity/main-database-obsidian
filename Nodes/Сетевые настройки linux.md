# Виртуальные интерфейсы
В Linux виртуальные сетевые интерфейсы представляют собой программные сущности, которые предоставляют системе те же возможности, что и физические сетевые карты, но реализованы полностью на уровне ядра. Они эмулируют поведение аппаратного обеспечения, позволяя операционной системе и приложениям взаимодействовать с ними стандартными сетевыми утилитами (ip, ifconfig)
## Типы виртуальных интерфейсов (GenByAI)
Все перечисленные ниже интерфейсы являются частью подсистемы `netdevice` ядра Linux [](https://dml.compkaluga.ru/forum/lofiversion/index.php/t107620.html). Их можно создавать и удалять стандартной утилитой `ip link` из пакета `iproute2`.

Механизм обработки пакетов для виртуальных интерфейсов интегрирован в стек ядра. При получении или отправке данных срабатывают зарегистрированные обработчики (например, через структуру `struct packet_type` или `rx_handler`), которые перенаправляют поток данных либо на другой интерфейс (как в случае с VETH или Bridge), либо в пользовательское пространство (TUN/TAP), либо производят дополнительные преобразования заголовков (VLAN, туннели).

# Базовые механизмы и структуры ядра (GenByAI)

Прежде чем описывать цепочки, важно определить общие элементы, участвующие в обработке:

- **socket buffer (`sk_buff`)**  
	Основной контейнер для пакета на всём пути через ядро. Содержит указатели на данные (L2, L3, L4 заголовки), метаданные (входящий/исходящий интерфейс `skb->dev`, набор флагов, временные метки) и указатель на результат маршрутизации (`dst_entry`). Пакеты передаются между подсистемами ядра путём передачи указателя на `sk_buff`.
	
- **NAPI (New API)**  
    Механизм взаимодействия драйвера сетевой карты с ядром. Совмещает прерывания и опрос (polling). При поступлении пакетов сетевая карта через DMA помещает их в кольцевой буфер (RX ring) и генерирует прерывание. Драйвер в обработчике прерывания отключает прерывания для этого устройства и ставит его в список опроса softIRQ. Далее в контексте `softirq` вызывается функция опроса драйвера, которая забирает пакеты из кольца, оборачивает их в `sk_buff` и передаёт в стек (`netif_receive_skb`). NAPI повышает производительность при высоком трафике.
    
- **Таблица маршрутизации (FIB — Forwarding Information Base)**  
    Хранит информацию об известных сетях и маршрутах. Представляет собой иерархическую структуру (обычно Trie или хеш-таблицы) для быстрого поиска по IP-адресу назначения (longest prefix match). Результат поиска — запись, содержащая выходной интерфейс, адрес следующего перехода (gateway), метрики и флаги. Кэш маршрутов (dst_entry) сохраняет результат для быстрого доступа.
    
- **Соседская таблица (Neighbour Table) — ARP/ND**  
    Хранит соответствие между IP-адресами L3 и MAC-адресами L2 для интерфейсов, подключенных к широковещательным сетям (Ethernet). Записи имеют состояние (например, `NUD_REACHABLE`, `NUD_STALE`, `NUD_INCOMPLETE`). При необходимости отправки пакета на прямой IP (в той же подсети) ядро обращается к этой таблице, чтобы получить MAC-адрес назначения. Если записи нет, инициируется ARP-запрос (или ND) и пакет ставится в очередь до получения ответа.
    
- **Таблица коммутации моста (FDB — Forwarding Database)**  
    Используется в сетевом мосте (bridge). Хранит пары {MAC-адрес, интерфейс-порт} для всех известных устройств за каждым портом моста. Таблица обучается на основе адресов источника проходящих кадров (learning) и используется для принятия решения о коммутации: если MAC назначения найден, кадр направляется только на соответствующий порт; если не найден или адрес широковещательный — кадр рассылается на все порты (flooding).
    
- **Netfilter хуки**  
    Набор точек внутри сетевого стека (например, PREROUTING, LOCAL_IN, FORWARD, POST_ROUTING для IPv4, а также аналогичные для bridge), где могут быть зарегистрированы функции обработки пакетов. Позволяют реализовать фильтрацию, NAT, изменение пакетов. Пакет, проходя через хук, может быть пропущен, изменён, отброшен или поставлен в очередь (например, в userspace через `nfqueue`).
    
- **RX/TX кольца (DMA-кольца)**  
    Области памяти, разделяемые с сетевой картой. Карта помещает принятые пакеты в RX-кольцо, используя прямой доступ к памяти (DMA), и забирает пакеты для отправки из TX-кольца. Драйвер оперирует дескрипторами в этих кольцах.
    
- **Очереди пакетов драйвера TUN/TAP**  
    Виртуальный драйвер `tun` (или `tap`) содержит кольцевые буферы (обычно `ptr_ring`) для передачи пакетов между ядром и пространством пользователя. Когда ядро отправляет пакет через TAP-интерфейс, драйвер помещает его в очередь, связанную с файловым дескриптором, который открыт в приложении (например, QEMU). Приложение читает из этого дескриптора, получая пакет. Обратное направление работает аналогично: запись приложения в дескриптор помещает пакет в очередь приёма драйвера, затем драйвер имитирует приход пакета из сети (`netif_rx`).
    
- **Сокеты (struct sock) и приёмные очереди**  
    Каждый открытый сетевой сокет (TCP, UDP, raw) имеет очередь входящих пакетов (`sk_receive_queue`). Когда пакет достигает транспортного уровня и идентифицируется как принадлежащий данному сокету (по IP-адресам и портам), он помещается в эту очередь. Приложение, выполняя системный вызов чтения (read, recvmsg), копирует данные из `sk_buff` в свой буфер.

### Виртуальные интерфейсы

- **TAP-интерфейс**  
    Виртуальное сетевое устройство, работающее на канальном уровне (L2). При отправке пакета через TAP ядро формирует полный Ethernet-кадр (с MAC-заголовком) и передаёт его драйверу. Драйвер доставляет этот кадр приложению, читающему из соответствующего файлового дескриптора. TAP используется, когда нужно эмулировать полноценный Ethernet-сегмент (например, для виртуальных машин, чтобы гость видел свою сетевую карту).
    
- **TUN-интерфейс**  
    Виртуальное устройство сетевого уровня (L3). Пакеты передаются без канального заголовка (начиная с IP). Используется для VPN (OpenVPN, WireGuard) и других туннелей, где не требуется эмуляция Ethernet.
    
- **Bridge (мост)**  
    Виртуальный коммутатор L2, объединяющий несколько сетевых интерфейсов (физических и виртуальных) в один логический сегмент. Каждый порт моста является полноценным сетевым устройством. Мост обрабатывает входящие кадры, принимает решение о пересылке на основе FDB и, при необходимости, доставляет кадры локальному стеку (если MAC назначения равен MAC самого моста).
    
- **vhost-net** (опционально, для QEMU)  
    Механизм ускорения виртуального ввода-вывода. Позволяет обойти лишние копирования между QEMU и ядром, передавая управление очередями напрямую из гостевого драйвера (virtio) в ядро хоста. В контексте нашего обсуждения можно рассматривать как оптимизацию, но базовая схема TAP + бридж остаётся основной.

### Итого
- **Маршрутизация (FIB)** определяет, на какой интерфейс отправить пакет и нужен ли следующий переход (gateway).
- **Соседская таблица (ARP/ND)** даёт MAC-адрес получателя в локальной сети.
- **FDB моста** решает, на какой порт отправить кадр внутри моста.
- **Таблицы сокетов** (по номерам портов) определяют, какому приложению принадлежит входящий пакет.
- **Очереди TUN/TAP** служат мостом между ядром и пользовательским пространством для виртуальных интерфейсов.
- **Netfilter** может изменять решения на любом из этапов.
# Пути прохождения пакетов (GenByAI)
### Сценарий 1: Пакет приходит на физический интерфейс и принимается локальным приложением (без виртуализации)

1. **Приём аппаратурой**  
    Сетевая карта получает кадр, проверяет FCS и, если адрес назначения совпадает с её MAC (или broadcast/multicast), сохраняет данные в RX-кольцо через DMA. Карта генерирует прерывание.
    
2. **Вход в ядро через NAPI**  
    Драйвер в обработчике прерывания регистрирует устройство для опроса. В softIRQ (`net_rx_action`) драйвер забирает пакеты из RX-кольца, для каждого создаёт `sk_buff`, заполняет поле `skb->dev` указателем на сетевое устройство, и вызывает `netif_receive_skb` — точку входа в стек.
    
3. **Обработка на канальном уровне**  
    Пакет проходит через точки подключения протоколов L2 (например, захватчики пакетов типа AF_PACKET). Если включён бриджинг, проверяется, не является ли устройство портом моста (но в данном случае нет). Затем определяется протокол L3 (обычно IP) и вызывается соответствующий обработчик.
    
4. **Обработка на сетевом уровне (IPv4)**  
    В обработчике `ip_rcv` проверяется контрольная сумма IP-заголовка. Пакет проходит через Netfilter хук **PREROUTING**. Затем вызывается функция маршрутизации, которая обращается к таблице FIB для поиска маршрута по IP-адресу назначения. Результат сохраняется в `skb_dst(skb)`. Для пакета, предназначенного локально, в поле `dst->input` устанавливается функция `ip_local_deliver`.
    
5. **Локальная доставка**  
    Пакет проходит через Netfilter хук **LOCAL_IN**. Затем вызывается обработчик транспортного уровня (например, `udp_rcv` или `tcp_v4_rcv`), который ищет сокет по адресам и портам в соответствующих хеш-таблицах. Если сокет найден, пакет помещается в его приёмную очередь (`sk_receive_queue`).
    
6. **Передача приложению**  
    Приложение, ожидающее данные (в цикле `epoll`/`select` или заблокированное на `read`), просыпается, когда в очереди сокета появляется пакет. Системный вызов чтения копирует данные из `sk_buff` в буфер пользователя, после чего `sk_buff` освобождается.

### Сценарий 2: Пакет уходит в TAP-интерфейс (например, виртуальной машине)

Рассмотрим направление от физической сети к виртуальной машине.

1. Пакет проходит шаги 1–3 (до определения L3) как в сценарии 1, но на этапе маршрутизации или бриджа решение другое. Допустим, TAP-интерфейс (`tap0`) включён в состав моста `br0` вместе с физическим интерфейсом `eth0`.
    
2. **Принятие решения мостом**  
    Пакет, пришедший на `eth0`, замечается механизмом бриджа. Он проверяет, является ли `eth0` портом моста. Если да, вызывается обработчик моста.
    
    - **Обучение (learning)**: MAC-адрес источника кадра сохраняется в FDB, связываясь с портом `eth0`.
        
    - **Поиск получателя**: по MAC-адресу назначения выполняется поиск в FDB. Если MAC найден и ассоциирован с портом `tap0`, кадр должен быть передан на `tap0`. Если MAC не найден или это broadcast, кадр рассылается на все порты, включая `tap0`.
        
3. **Передача на TAP-интерфейс**  
    Кадр (в виде `sk_buff`) направляется драйверу устройства `tap0`. Драйвер TAP (в `drivers/net/tun.c`) получает кадр через функцию отправки. Поскольку устройство виртуальное, драйвер не передаёт его в физическое кольцо, а помещает `sk_buff` в приёмную очередь (кольцевой буфер), связанную с файловым дескриптором, открытым в приложении (QEMU). Если очередь заполнена, пакет может быть отброшен (или драйвер может остановить очередь, создавая обратное давление).
    
4. **Передача в пространство пользователя**  
    Приложение (QEMU), выполняющее `read` на `/dev/net/tun`, находится в состоянии ожидания. Когда пакет появляется в очереди, драйвер пробуждает приложение. QEMU копирует данные кадра в свой буфер (в адресное пространство пользователя). Затем QEMU передаёт этот кадр гостевой системе через эмулируемую сетевую карту (например, virtio-net или e1000), используя механизмы виртуализации (например, запись в разделяемую память или генерацию виртуального прерывания).

### Сценарий 3: Пакет проходит через мост (bridge) без локальной доставки

Рассмотрим кадр, пришедший на один порт моста и переданный на другой физический порт (коммутация L2).

1. Кадр поступает на `eth0`, который является портом моста `br0`. Обработчик моста выполняет learning (сохраняет источник MAC в FDB с пометкой порта `eth0`).
    
2. Поиск в FDB по MAC-адресу назначения даёт порт `eth1` (например). Тогда кадр должен быть передан на `eth1`.
    
3. **Обработка перед отправкой**  
    Кадр может проходить через Netfilter хуки моста (например, `NF_BR_FORWARD`). Если фильтрация разрешает проход, кадр передаётся драйверу `eth1` через вызов `dev_queue_xmit`.
    
4. **Отправка через физический интерфейс**  
    Драйвер `eth1` помещает кадр в TX-кольцо и уведомляет карту о наличии данных для отправки. Карта передаёт кадр в физическую среду.
    

Если кадр имеет broadcast-адрес, он копируется на все порты моста, кроме входящего, включая локальный стек (если мост имеет IP-адрес и кадр проходит фильтрацию).

### Сценарий 4: Пакет уходит через TUN-интерфейс (VPN)

Рассмотрим отправку пакета от локального приложения на хост, который маршрутизируется в VPN.

1. Приложение отправляет дейтаграмму UDP (например) на удалённый адрес в VPN-сети. Сокет передаёт пакет в IP-стек.
    
2. **Маршрутизация**  
    Ядро выполняет поиск в таблице FIB. Найденный маршрут указывает, что пакет следует отправлять через интерфейс `tun0` (созданный VPN-клиентом). Результат сохраняется в `dst_entry`.
    
3. **Передача драйверу TUN**  
    Вызывается функция отправки драйвера `tun`. Так как это TUN (а не TAP), драйвер ожидает пакет без Ethernet-заголовка (начиная с IP). Драйвер помещает `sk_buff` в очередь, связанную с файловым дескриптором VPN-клиента.
    
4. **Получение приложением VPN**  
    VPN-клиент читает пакет из дескриптора, шифрует его и отправляет через физический сокет на реальный VPN-сервер.
    
5. **Обратное направление (ответ от сервера)**  
    Зашифрованный ответ приходит на физический интерфейс, поднимается до сокета VPN-клиента. Клиент расшифровывает пакет и записывает полученный IP-пакет в файловый дескриптор TUN. Драйвер TUN создаёт `sk_buff` с данными пакета, устанавливает `skb->dev = tun0`, и вызывает `netif_rx` (или `napi_gro_receive`), имитируя приход пакета из сети. Далее пакет обрабатывается как обычный входящий: IP-стек, поиск маршрута (теперь адрес назначения локальный) и доставка в исходный сокет приложения.

### Сценарий 5: Пакет из гостевой системы QEMU выходит в физическую сеть (через TAP + бридж)

Рассмотрим типичную настройку: QEMU с `-netdev tap,id=net0 -device virtio-net,netdev=net0`. Интерфейс `tap0` добавлен в мост `br0` вместе с физическим `eth0`.

1. **Гостевая система отправляет кадр**  
    Гостевой драйвер virtio-net помещает кадр в виртуальное кольцо (vring). QEMU как VMM обрабатывает это кольцо (или vhost-net обрабатывает напрямую). QEMU получает кадр и записывает его в файловый дескриптор TAP (`write` на `/dev/net/tun`). Данные содержат полный Ethernet-кадр (с MAC-адресами гостя и шлюза).
    
2. **Приём в ядро хоста через TAP**  
    Драйвер TUN/TAP получает данные, создаёт `sk_buff`, заполняет `skb->dev = tap0` и передаёт в стек через `netif_rx` (имитация прихода пакета).
    
3. **Обработка мостом**  
    Поскольку `tap0` — порт моста `br0`, пакет попадает в обработчик моста. Мост выполняет learning (MAC-адрес источника — гостевой MAC — связывается с портом `tap0`). Затем по MAC-адресу назначения (шлюз, например) выполняется поиск в FDB. Если MAC шлюза ранее был изучен за портом `eth0`, кадр направляется на `eth0`.
    
4. **Выход через физический интерфейс**  
    Пакет передаётся драйверу `eth0` и отправляется в физическую сеть через DMA.
    
5. **Ответ от внешнего сервера**  
    Ответный кадр приходит на `eth0`. Мост, сверившись с FDB, видит, что MAC-адрес гостя связан с `tap0`, и пересылает кадр на `tap0`. Драйвер TUN помещает его в очередь для QEMU. QEMU читает кадр и передаёт его гостю через virtio-кольцо.

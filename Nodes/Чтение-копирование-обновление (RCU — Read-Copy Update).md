**Read-Copy Update (RCU)** — это механизм синхронизации в ядре Linux, предназначенный для ситуаций с преобладанием операций чтения над операциями записи (read-mostly data structures). Основная цель RCU — позволить нескольким читателям выполняться параллельно с писателем без использования традиционных блокировок, таких как спин-локи (spinlocks) или читатель-писательские блокировки (rwlocks), которые могут создавать конкуренцию и приводить к простоям кэша [](https://www.eklektix.com/Articles/371986/)[](https://www.read.seas.harvard.edu/cs161/2019/lectures/lecture18/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).

### 1. Принцип работы: копирование при записи и отложенное освобождение

Ключевая идея RCU заключается в разграничении доступа к данным на три фазы: удаление (removal), ожидание (grace period) и перезапись (reclamation). Это позволяет читателям никогда не блокироваться, но требует от писателя выполнения дополнительных действий для обеспечения целостности данных [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).

Процесс обновления данных, защищенных RCU, можно описать следующей последовательностью шагов (на примере обновления элемента массива или указателя):

1. **Чтение (Read):** Несколько потоков (читателей) параллельно входят в критическую секцию чтения (например, с помощью `rcu_read_lock()`) и получают доступ к данным по текущему указателю. В этот момент они работают со "старой" версией данных. Критическая секция чтения не является блокировкой в классическом смысле; в ядре Linux она часто реализуется просто как запрет вытеснения (preemption disable) [](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
2. **Копирование (Copy):** Когда появляется писатель (поток, желающий изменить данные), он не модифицирует существующие данные напрямую. Вместо этого он создает новую копию этих данных (новую версию). Для этого может быть выделена новая память (например, `kmalloc`) и скопировано содержимое старого объекта [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
3. **Обновление (Update):** Писатель модифицирует данные в созданной копии.
    
4. **Публикация (Publish):** Писатель атомарно обновляет глобальный указатель (публикует новую версию), используя специальный примитив, такой как `rcu_assign_pointer()`. Этот макрос гарантирует, что все предыдущие операции записи (инициализация новой копии) будут видны всем остальным процессорам до того, как будет обновлен указатель [](https://www.eklektix.com/Articles/371986/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
5. **Разделение читателей (Partitioning):** Начиная с этого момента, любые _новые_ читатели, входящие в критическую секцию, будут видеть уже новую версию данных (так как они обращаются по обновленному указателю). _Старые_ читатели, которые уже вошли в критическую секцию _до_ публикации, продолжают работать со старой версией данных, так как они сохранили указатель на нее.
    
6. **Ожидание (Grace Period):** Писатель не может сразу освободить память, занятую старой версией, так как старые читатели всё еще могут на нее ссылаться. Он должен дождаться так называемого "периода благодати" (Grace Period) — промежутка времени, в течение которого все существовавшие на момент публикации читатели гарантированно завершат свои критические секции [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
7. **Перезапись (Reclamation):** Как только все процессоры (или задачи, в зависимости от реализации RCU) зафиксируют прохождение через состояние покоя (quiescent state), что сигнализирует о завершении всех старых критических секций, период благодати заканчивается. После этого писатель (или специальный callback) может безопасно освободить память старой версии, например, с помощью `kfree()` или через callback, зарегистрированный `call_rcu()` [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    

### 2. Ключевые понятия и терминология

- **Критическая секция чтения (Read-Side Critical Section):** Область кода между `rcu_read_lock()` и `rcu_read_unlock()`. Читатели должны находиться в этой секции для доступа к данным, защищенным RCU. В этот период читатель гарантированно видит консистентное состояние данных и защищен от их преждевременного удаления [](https://www.eklektix.com/Articles/371986/)[](https://www.programmersought.com/article/11022727474/).
    
- **Период благодати (Grace Period):** Временной промежуток, необходимый для того, чтобы все читатели, активные на момент начала этого периода, завершили свои критические секции. RCU гарантирует, что любой объект, удаленный из общего доступа до начала периода благодати, будет безопасно освобожден только после его окончания [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
- **Состояние покоя (Quiescent State):** Момент времени, когда текущий поток (или процессор) не находится внутри критической секции RCU. Примером может служить переключение контекста (context switch) или выполнение кода в пользовательском пространстве. RCU считает, что все читатели, начавшие работу до определенного момента, завершились, если каждый процессор прошел хотя бы через одно состояние покоя [](https://www.programmersought.com/article/11022727474/)[](https://en.eeworld.com.cn/mp/rrgeek/a118100.jspx).
    
- **Примитивы публикации и подписки (Publish/Subscribe):** Для обеспечения корректности работы на архитектурах со слабой моделью памяти используются специальные макросы. `rcu_assign_pointer()` используется писателем для "публикации" нового указателя. `rcu_dereference()` используется читателем для безопасного получения указателя внутри критической секции, гарантируя, что все связанные с ним данные будут видны [](https://www.eklektix.com/Articles/371986/)[](https://www.programmersought.com/article/11022727474/).
    

### 3. RCU и структура `task_struct`

В ядре Linux структура `task_struct`, описывающая каждый процесс, является классическим примером использования RCU. Список процессов (`task_struct`) защищен именно этим механизмом [](https://gitlab.arm.com/linux-arm/linux-iv/-/blob/150593bf869393d10a79f6bd3df2585ecc20a9bb/include/linux/sched.h)[](https://glab.starfivetech.com/jh7110/jh7110-visionfive2/linux/-/commit/8c7904a00b06d2ee51149794b619e07369fcf9d4).

- **Защита от прямого изменения:** Несмотря на то, что код выполняется на уровне ядра, прямое изменение полей `task_struct` без использования RCU-примитивов недопустимо. Хотя такие изменения могут быть временно видны, они не будут безопасны с точки зрения синхронизации и могут привести к повреждению данных или обращению к уже освобожденной памяти.
    
- **Поиск процессов:** Для поиска процесса по его идентификатору (PID) используется идиома, гарантирующая безопасность:

```c unfold
   rcu_read_lock();
    task = find_task_by_vpid(pid); // Доступ к списку, защищенному RCU
    if (task)
        get_task_struct(task); // Увеличиваем счетчик ссылок, чтобы задача не исчезла
    rcu_read_unlock();
    // ... работа с task ...
    put_task_struct(task); // Уменьшаем счетчик, когда задача больше не нужна
```

* В этом примере RCU защищает обход списка и доступ к указателю на `task_struct`. Поскольку поиск происходит внутри `rcu_read_lock()`, структура `task_struct` не может быть освобождена до вызова `rcu_read_unlock()`. Однако, если планируется работа с задачей после выхода из RCU-секции, необходимо увеличить счетчик ссылок с помощью `get_task_struct()` [](https://gitlab.arm.com/linux-arm/linux-iv/-/blob/150593bf869393d10a79f6bd3df2585ecc20a9bb/include/linux/sched.h)[](https://glab.starfivetech.com/jh7110/jh7110-visionfive2/linux/-/commit/8c7904a00b06d2ee51149794b619e07369fcf9d4).
  
- **Отложенное освобождение:** Освобождение памяти `task_struct` происходит не сразу после вызова `put_task_struct()`, когда счетчик ссылок падает до нуля. Вместо этого, функция `release_task()` помещает структуру в список на удаление, и фактическое освобождение памяти делегируется callback-функции через `call_rcu()`, гарантируя, что ни один читатель, все еще держащий указатель, не обратится к уже освобожденной памяти [](https://glab.starfivetech.com/jh7110/jh7110-visionfive2/linux/-/commit/8c7904a00b06d2ee51149794b619e07369fcf9d4).
    

### 4. Сравнение с пользовательским пространством и C++

В стандартную библиотеку C++ (STL) механизм RCU не входит. Однако существуют пользовательские реализации, такие как `liburcu` (userspace RCU), которые предоставляют схожую функциональность для высоконагруженных приложений с интенсивным чтением [](https://github.com/colinjfw/read-copy-update)[](https://docs.rs/crate/read-copy-update/0.1.0)[](https://www.php.cn/faq/1910212.html). В ядре Linux RCU является зрелым и активно развивающимся механизмом, интегрированным в подсистемы управления памятью и планировщик задач [](https://www.eklektix.com/Articles/371986/).

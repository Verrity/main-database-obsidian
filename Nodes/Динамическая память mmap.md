### 1. Сущность `mmap`: Работа с дескрипторами и VMA

Системный вызов `mmap` (memory map) служит для создания нового отображения в виртуальном адресном пространстве процесса [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html). Грубо говоря, он создает новую область виртуальной памяти (VMA — Virtual Memory Area).

**Прототип и базовая механика:**

```c unfold
#include <sys/mman.h>
void *mmap(void *addr, size_t length, int prot, int flags, int fd, off_t offset);
```

- **`addr`**: Желаемый адрес размещения (hint). Если передать `NULL`, ядро само подберет выровненный по странице адрес — это самый портируемый способ [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    
- **`length`**: Размер отображения в байтах (должен быть больше 0).
    
- **`prot`**: Защита памяти (например, `PROT_READ | PROT_WRITE`). Важно: она не должна конфликтовать с режимом открытия файла.
    
- **`flags`**: Самая важная часть, определяющая семантику (разделяемый/частный, анонимный/файловый).
    
- **`fd` и `offset`**: Дескриптор файла и смещение в нем. Если отображение анонимное (`MAP_ANONYMOUS`), `fd` игнорируется (нужно передавать -1) [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    

### 2. Типы отображений (`flags`)

В `flags` мы определяем, _как_ именно данные будут синхронизироваться с диском и будут ли видны другим.

- **`MAP_SHARED`**:
    
    - Изменения в памяти видимы другим процессам, которые отобразили тот же файл.
        
    - Изменения в итоге записываются в сам файл (хотя для гарантии записи на диск требуется `msync(2)`) [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
        
    - Отличный механизм для межпроцессного взаимодействия (IPC) через разделяемую память.
        
- **`MAP_PRIVATE`**:
    
    - Создает private mapping с механизмом **copy-on-write (CoW)**.
        
    - Изначально все процессы читают одни и те же физические страницы (page cache).
        
    - Как только процесс пытается писать, ядро создает приватную копию страницы для этого процесса. Изменения не видны другим и не уходят обратно в файл [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html). Типичный пример — загрузка исполняемых файлов и библиотек.
        
- **`MAP_ANONYMOUS`**:
    
    - Отображение не привязано к файлу (флаг `MAP_ANON`). `fd` игнорируется, содержимое инициализируется нулями [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html). Используется для аллокации большой приватной памяти (например, в реализации `malloc`).
        

### 3. Файловый дескриптор и "Упреждающее чтение" (Read-Ahead)

Исходный тезис верен: `mmap` работает с файловыми дескрипторами, и на него распространяется политика буфера упреждающего чтения. Однако здесь есть важный нюанс, который мы обязаны добавить: эвристика **`mmap_miss`**.

Ядро Linux пытается оптимизировать производительность при последовательном доступе к отображенному файлу. Когда процесс впервые обращается к странице, возникает страничная ошибка (page fault). В обработчике `do_async_mmap_readahead` (или `do_sync_mmap_readahead`) срабатывает следующий алгоритм:

1. У каждого отображения (VMA) есть счетчик `ra->mmap_miss`.
    
2. Если страница уже была в кеше (page cache) — это "попадание" (cache hit). Счетчик `mmap_miss` уменьшается.
    
3. Если страницы в кеше не было — это "промах" (cache miss).
    
4. Если `mmap_miss` падает ниже определенного порога, ядро включает механизм упреждающего чтения: при промахе оно подгружает в память не одну страницу, а сразу несколько (синхронно или асинхронно), предполагая, что процесс будет читать файл последовательно [](https://lkml.org/lkml/2025/7/10/1623).
    

**Однако, в современных ядрах (обсуждения в LKML 2025 года [](https://lkml.org/lkml/2025/7/10/1623)[](https://lkml.org/lkml/2025/7/28/393)) в эту логику внесены важные изменения для борьбы с "трешем" (thrashing):**

Раньше множественные треды, фолтящие на одну и ту же страницу (особенно под давлением памяти), могли создать "порочный круг": они генерировали много minor page faults, что агрессивно уменьшало `mmap_miss` и заставляло ядро включать read-ahead, хотя памяти и так не хватало. Это приводило к еще большей нагрузке.

Согласно патчам, теперь при проверке `do_async_mmap_readahead` ядро смотрит на флаги `folio` (структура страницы):

- Если `folio_test_workingset(folio)` истинно, это означает, что страница была вытеснена (evicted) из памяти и теперь загружается заново. Это признак "треша".
    
- В ответ на это ядро не уменьшает `mmap_miss`, а наоборот, **увеличивает** его (например, добавляя `MMAP_LOTSAMISS / 2`), тем самым заставляя read-ahead отключиться [](https://lkml.org/lkml/2025/7/10/1623).
    
- Также рассматривается вариант не уменьшать `mmap_miss`, если страница заблокирована (folio locked), что указывает на конкурентный фолт [](https://lkml.org/lkml/2025/7/28/393).
    

**Вывод по тезису:** `mmap` действительно работает с файловыми дескрипторами и подчиняется read-ahead. Но read-ahead в Linux — это динамическая эвристика, которая может отключаться (или даже инвертировать поведение) при обнаружении признаков нехватки памяти (thrashing), чтобы не усугублять ситуацию.

### 4. Детали реализации: Структура `vm_area_struct`

Каждое отображение, созданное через `mmap`, представлено в ядре структурой `struct vm_area_struct`. Это ключ к пониманию того, как ядро "видит" нашу память [](https://cloud.tencent.com.cn/developer/article/2352778?from=15425&frompage=seopage).

```c unfold
struct vm_area_struct {
    unsigned long vm_start;   // Начало в виртуальном адресном пространстве
    unsigned long vm_end;     // Конец
    struct file * vm_file;    // Указатель на отображенный файл (если есть)
    unsigned long vm_pgoff;   // Смещение в файле (в страницах)
    // ... другие поля: защита, флаги и т.д.
}
```

Все VMA процесса организованы в красно-черное дерево и двусвязный список для эффективного поиска и перебора [](https://cloud.tencent.com.cn/developer/article/2352778?from=15425&frompage=seopage).

### 5. Пример кода (Файловый `MAP_PRIVATE`)

Ниже пример, демонстрирующий чтение файла через `mmap`. Здесь мы видим стандартную последовательность: `open`, `fstat` (чтобы узнать длину), `mmap`, доступ к памяти, `munmap` [](https://www.tencentcloud.com/techpedia/106433).

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/mman.h>
#include <sys/stat.h>
#include <fcntl.h>
#include <unistd.h>
#include <string.h>
int main() {
    int fd = open("example.txt", O_RDWR);
    if (fd == -1) {
        perror("open");
        exit(EXIT_FAILURE);
    }
    struct stat sb;
    if (fstat(fd, &sb) == -1) {
        perror("fstat");
        close(fd);
        exit(EXIT_FAILURE);
    }
    // Длина файла не должна быть 0 для данного примера
    if (sb.st_size == 0) {
        fprintf(stderr, "File is empty\n");
        close(fd);
        exit(EXIT_FAILURE);
    }
    // Создаем приватное отображение только для чтения
    char *addr = mmap(NULL, sb.st_size, PROT_READ, MAP_PRIVATE, fd, 0);
    if (addr == MAP_FAILED) {
        perror("mmap");
        close(fd);
        exit(EXIT_FAILURE);
    }
    // Файловый дескриптор можно сразу закрыть после mmap.
    // Отображение продолжает существовать.
    close(fd);
    // Работа с данными как с обычным массивом в памяти
    printf("File content:\n%.*s\n", (int)sb.st_size, addr);
    // Попытка записи сюда вызвала бы Segmentation Fault из-за PROT_READ
    if (munmap(addr, sb.st_size) == -1) {
        perror("munmap");
        exit(EXIT_FAILURE);
    }
    return 0;
}
```
### 6. Особые случаи и best practice

1. **Выравнивание**: `offset` и `addr` (если указан явно) должны быть кратны размеру страницы (обычно 4K). В Linux это `sysconf(_SC_PAGE_SIZE)` [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    
2. **Флаг `MAP_FIXED`**: Крайне опасен. Он говорит ядру: "ставь отображение точно сюда, даже если уничтожишь существующие маппинги". Используй `MAP_FIXED_NOREPLACE` (Linux 4.17+), чтобы проверить, свободен ли адрес [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    
3. **Синхронизация**: Для `MAP_SHARED` изменения не попадают на диск мгновенно. Используй `msync(2)` с флагом `MS_SYNC` или `MS_ASYNC` для гарантированной записи [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    
4. **Большие страницы (HugeTLB)**: Для высоконагруженных приложений можно использовать флаги `MAP_HUGETLB` в сочетании с `MAP_HUGE_2MB` или `MAP_HUGE_1GB`, чтобы уменьшить количество уровней трансляции адресов (TLB misses) [](https://manpages.opensuse.org/Leap-15.6/man-pages-ru/mmap.2.ru.html).
    
5. **Advice**: После создания маппинга можно дать ядру "совет" с помощью `madvise(2)`. Например, `MADV_RANDOM` скажет ядру отключить read-ahead для данного диапазона, а `MADV_SEQUENTIAL` — наоборот, агрессивно читать вперед [](https://lab.nexedi.com/kirr/wendelin.core/-/commit/cdde581d977ee9d2206f9709528c14eb11a29384).
    

### Резюме

Изначальный тезис — "MMAP работает с дескрипторами файлов и на него распространяется политика буфера упреждающего чтения" — верен, но является лишь вершиной айсберга.

Полная картина такова: `mmap` — это механизм создания VMA, связывающий файловый дескриптор (через структуру `file`) с диапазоном виртуальных адресов процесса. Упреждающее чтение (read-ahead) в контексте `mmap` управляется сложной эвристикой, базирующейся на счетчике `mmap_miss` и флагах страниц (таких как `workingset`), и может динамически отключаться при признаках "треша" (thrashing) для стабилизации системы

## Принцип работы
### 1. Механизм `select`

`select` - это исторически первый и самый переносимый (POSIX-совместимый) системный вызов для мультиплексирования.

#### Системный вызов и API

```c unfold
#include <sys/select.h>
int select(int nfds, fd_set *readfds, fd_set *writefds,
           fd_set *exceptfds, struct timeval *timeout);
FD_ZERO(fd_set *set);
FD_SET(int fd, fd_set *set);
FD_CLR(int fd, fd_set *set);
FD_ISSET(int fd, fd_set *set); // Макрос для проверки в User Space после возврата
```

#### Принцип работы (User Space Perspective)

1. **Подготовка**: Приложение инициализирует три битовые маски (`fd_set`): для чтения, записи и исключений. С помощью макроса `FD_SET` устанавливаются биты, соответствующие интересующим файловым дескрипторам.
    
2. **Вызов**: Происходит системный вызов `select`. Это точка входа в ядро.
    
3. **Ожидание и возврат**: Ядро блокирует процесс до тех пор, пока хотя бы один из отслеживаемых дескрипторов не станет готов к операции, либо не истечет таймаут. После пробуждения `select` возвращает управление в User Space, модифицируя переданные маски (`readfds`, `writefds`, `exceptfds`), чтобы они указывали только на готовые дескрипторы.
    
4. **Обработка результата**: В User Space приложение циклически проходит по всем интересующим его дескрипторам (от 0 до `nfds-1`), используя макрос `FD_ISSET`, чтобы определить, какие из них готовы, и выполняет соответствующие операции ввода-вывода.
    

#### Анализ производительности и ограничения (с точки зрения реализации в ядре)

Несмотря на простоту API, механизм `select` имеет ряд фундаментальных недостатков, которые проявляются при росте количества отслеживаемых дескрипторов (до десятков тысяч):

- **Ограничение на максимальный номер дескриптора**: Значение `nfds` должно быть равно `максимальный_номер_дескриптора + 1`. Сами маски имеют фиксированный размер `FD_SETSIZE` (обычно 1024). Это означает, что `select` не может эффективно работать с дескрипторами, номера которых больше 1023. Формально этот лимит можно обойти перекомпиляцией ядра, но это не является стандартной практикой [](https://developer.aliyun.com/article/1123953)[](https://pingkai.cn/article/post/4473.html).
    
- **Копирование битовых масок**: При каждом вызове `select` ядро вынуждено копировать все три битовые маски (`fd_set`) из User Space в Kernel Space. Для большого количества неактивных дескрипторов это является напрасной тратой ресурсов и времени [](https://developer.aliyun.com/article/1123953)[](https://github.com/Liu-YT/IO-Multiplexing).
    
- **Линейное сканирование (O(n))**: После копирования ядро выполняет линейный обход (polling) всех переданных дескрипторов (от 0 до `nfds-1`), проверяя состояние каждого. Время работы алгоритма линейно зависит от максимального номера дескриптора `nfds`, а не от количества активных соединений [](https://developer.aliyun.com/article/1123953)[](https://ru.wikipedia.org/wiki/Epoll).
    
- **Модификация и обратное копирование**: После пробуждения ядро модифицирует переданные маски, после чего они копируются обратно в User Space. Это еще одна операция копирования.

### 2. Механизм `poll`

Системный вызов `poll` был разработан как улучшение `select`, устраняющее некоторые его ограничения.

#### Системный вызов и API

```c unfold
#include <poll.h>
int poll(struct pollfd *fds, nfds_t nfds, int timeout);
struct pollfd {
    int   fd;      /* файловый дескриптор */
    short events;  /* запрошенные события (битовая маска, устанавливается в User Space) */
    short revents; /* возвращенные события (заполняется ядром) */
};
```

#### Принцип работы (User Space Perspective)

1. **Подготовка**: Приложение создает массив структур `struct pollfd`. Для каждого интересующего дескриптора заполняется поле `fd` и поле `events` (битовая маска событий, например, `POLLIN`).
    
2. **Вызов**: Приложение передает ядру указатель на массив `fds` и его размер `nfds`.
    
3. **Ожидание и возврат**: Ядро блокирует процесс до наступления события или таймаута. В отличие от `select`, ядро не перезаписывает поле `events`, а заполняет отдельное поле `revents` для каждого дескриптора, по которому произошло событие.
    
4. **Обработка результата**: В User Space приложение итерируется по массиву `fds` и проверяет значение поля `revents` у каждого элемента. Ненулевое значение `revents` указывает на готовность дескриптора.
    

#### Анализ производительности (User Space vs Kernel Space)

`poll` решил проблему фиксированного размера масок:

- **Нет ограничения на максимальный номер fd**: `poll` работает с любыми файловыми дескрипторами, так как использует массив структур, а не битовую маску фиксированной длины [](https://developer.aliyun.com/article/1123953)[](https://pingkai.cn/article/post/4473.html).
    
- **Разделение событий**: Разделение полей `events` и `revents` избавляет от необходимости пересобирать массив запросов перед каждым вызовом, как это было с `fd_set` в `select`.
    

Однако фундаментальные проблемы производительности остались:

- **Копирование всего массива**: При каждом вызове `poll` ядро по-прежнему копирует весь массив структур `pollfd` из User Space в Kernel Space. Размер этого массива может достигать сотен тысяч элементов [](https://pingkai.cn/article/post/4473.html).
    
- **Линейное сканирование (O(n))**: Ядро вынуждено линейно проходить по всему массиву `fds` (размером `nfds`) для проверки состояния каждого дескриптора. Это сохраняет линейную сложность O(n) [](https://ru.wikipedia.org/wiki/Epoll)[](https://github.com/Liu-YT/IO-Multiplexing).

### 3. Механизм `epoll`

`epoll` - это специфичный для Linux механизм, разработанный для преодоления ограничений масштабируемости `select` и `poll`. Он работает по событийно-ориентированной модели.

#### Системные вызовы (Трехступенчатый API)

```c unfold
#include <sys/epoll.h>
// 1. Создание контекста epoll
int epoll_create(int size); // Устаревший, size игнорируется, но должен быть > 0
int epoll_create1(int flags); // Современная версия с флагами (например, EPOLL_CLOEXEC)
// 2. Управление списком интересов (epoll set)
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
// 3. Ожидание событий
int epoll_wait(int epfd, struct epoll_event *events, int maxevents, int timeout);
```
#### Принцип работы (User Space Perspective)

1. **Создание контекста (`epoll_create`)**: Приложение просит ядро создать структуру данных, называемую _экземпляром epoll_ (epoll instance). Ядро возвращает обычный файловый дескриптор (`epfd`), который используется для всех последующих операций с этим экземпляром [](https://manpages.debian.org/bookworm/manpages/epoll.7.en.html)[](https://ru.wikipedia.org/wiki/Epoll)[](https://www.man.he.net/man7/epoll).
    
2. **Управление интересом (`epoll_ctl`)**: Это ключевое отличие. Вместо того чтобы каждый раз передавать ядру полный список дескрипторов, приложение однократно регистрирует (или удаляет/изменяет) дескрипторы, которые его интересуют, с помощью `epoll_ctl`. При регистрации передается структура `struct epoll_event`, содержащая битовую маску интересующих событий и пользовательские данные (чаще всего сам файловый дескриптор) [](https://developer.aliyun.com/article/1123953)[](https://ru.wikipedia.org/wiki/Epoll).
    
3. **Ожидание событий (`epoll_wait`)**: Приложение вызывает `epoll_wait`, передавая массив `events` достаточно большого размера. Ядро заполняет этот массив информацией только о _реально готовых_ (ready) дескрипторах. Количество готовых дескрипторов возвращается в качестве результата функции.
    

#### Анализ архитектуры и производительности (O(1))

Производительность `epoll` обусловлена его внутренней структурой в ядре, которая концептуально состоит из двух основных списков [](https://manpages.debian.org/bookworm/manpages/epoll.7.en.html)[](https://www.man.he.net/man7/epoll)[](https://github.com/Liu-YT/IO-Multiplexing):

- **Список интересов (Interest List)**: Хранится в ядре. Для быстрого поиска и управления (при вызовах `epoll_ctl`) часто реализуется с помощью красно-черного дерева. Здесь хранятся все дескрипторы, зарегистрированные приложением.
    
- **Готовый список (Ready List)**: Двусвязный список, содержащий ссылки на те дескрипторы из списка интересов, на которых произошли события.
    

**Как это работает в ядре:**

1. **Регистрация (`epoll_ctl`)**: Когда дескриптор добавляется в интерес лист, ядро устанавливает для него колбэк (callback-функцию). Этот колбэк срабатывает, когда с дескриптором происходит событие (например, приходят сетевые данные).
    
2. **Наступление события**: При возникновении события на дескрипторе (например, в сетевом драйвере) ядро вызывает ранее установленный колбэк. Задача колбэка — добавить структуру, описывающую этот дескриптор, в готовый список (ready list) соответствующего экземпляра `epoll`.
    
3. **Пробуждение (`epoll_wait`)**: Когда приложение вызывает `epoll_wait`, ядро просто проверяет, пуст ли готовый список. Если список не пуст, ядро копирует содержимое готового списка (ограниченное параметром `maxevents`) в массив `events` в User Space. Если список пуст, процесс переводится в режим ожидания до появления события в списке.
    

**Ключевые преимущества:**

- **Отсутствие массового копирования**: Копирование данных из User Space в Kernel Space происходит только один раз, на этапе регистрации дескриптора через `epoll_ctl`. Последующие вызовы `epoll_wait` копируют обратно в User Space лишь небольшой массив готовых событий [](https://developer.aliyun.com/article/1123953)[](https://github.com/Liu-YT/IO-Multiplexing).
    
- **Отсутствие линейного сканирования**: Ядру не нужно сканировать тысячи дескрипторов. Оно работает по принципу "call me when you're ready" (позови меня, когда будешь готов). Сложность получения списка готовых событий составляет O(1) относительно общего количества отслеживаемых дескрипторов [](https://ru.wikipedia.org/wiki/Epoll)[](https://github.com/Liu-YT/IO-Multiplexing).
    
- **Масштабируемость**: `epoll` без проблем работает с десятками тысяч файловых дескрипторов (лимит определяется только объемом оперативной памяти и настройками `max_user_watches` в `/proc` [](https://manpages.debian.org/bookworm/manpages/epoll.7.en.html)[](https://ru.wikipedia.org/wiki/Epoll)).
    

#### Режимы работы: Level-Triggered (LT) и Edge-Triggered (ET)

`epoll` может работать в двух режимах, что дает разработчику тонкий контроль над поведением [](https://manpages.debian.org/bookworm/manpages/epoll.7.en.html)[](https://ru.wikipedia.org/wiki/Epoll)[](https://pingkai.cn/article/post/4473.html):

- **Level-Triggered (LT, по умолчанию)**:
    
    - **Поведение**: Если в буфере дескриптора остались непрочитанные данные, `epoll_wait` будет _продолжать возвращать_ этот дескриптор при каждом вызове до тех пор, пока данные не будут полностью считаны.
        
    - **Аналогия**: Работает как `poll` или `select`. Проще в использовании, менее подвержен ошибкам.
        
- **Edge-Triggered (ET, режим высокой производительности)**:
    
    - **Поведение**: Событие о готовности дескриптора генерируется _только один раз_ в момент изменения состояния (например, когда в пустой буфер поступили новые данные). Если приложение считало не все данные, новый вызов `epoll_wait` _не вернет_ этот дескриптор до тех пор, пока не поступят _новые_ данные.
        
    - **Требования**: Требует использования неблокирующих (non-blocking) дескрипторов и циклического чтения/записи до получения ошибки `EAGAIN` или `EWOULDBLOCK`. Это позволяет избежать блокировки и дает максимальную производительность, так как минимизирует количество системных вызовов.
        
    - **Многопоточность**: ET-режим также гарантирует, что при готовности дескриптора будет разбужен только один поток из нескольких, ожидающих на одном экземпляре `epoll`, что предотвращает проблему "thundering herd" [](https://manpages.debian.org/bookworm/manpages/epoll.7.en.html)[](https://www.man.he.net/man7/epoll).
        

### Сравнительная таблица

|Характеристика|select|poll|epoll|

|**Переносимость**|POSIX (везде)|POSIX (везде)|Только Linux|
|**Режимы**|Только уровень|Только уровень|Уровень (LT) и Край (ET)|

### Заключение

С точки зрения разработчика, эволюция от `select` к `epoll` — это переход от пассивного опроса (polling) к активному уведомлению (event-driven). `select` и `poll` просты, но их производительность деградирует линейно с ростом числа соединений из-за необходимости копировать большие структуры данных и сканировать их в ядре. `epoll` решает эти проблемы, предоставляя масштабируемый механизм, где ядро берет на себя работу по отслеживанию состояния и уведомляет приложение только о готовых дескрипторах, что делает его стандартом де-факто для высокопроизводительных сетевых приложений в Linu

## Примеры
### 1. Пример использования `select`

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/select.h>
#include <unistd.h>
#include <string.h>

int main() {
    fd_set readfds;
    struct timeval tv;
    int retval;
    // Очищаем набор дескрипторов
    FD_ZERO(&readfds);
    // Добавляем stdin (fd 0) в набор для чтения
    FD_SET(STDIN_FILENO, &readfds);
    // Устанавливаем тайм-аут: 5 секунд
    tv.tv_sec = 5;
    tv.tv_usec = 0;
    // Вызываем select. Первый аргумент - максимальный номер дескриптора + 1.
    // Возвращает количество готовых дескрипторов, 0 при тайм-ауте, -1 при ошибке.
    retval = select(STDIN_FILENO + 1, &readfds, NULL, NULL, &tv);
    if (retval == -1) {
        perror("select()");
        exit(EXIT_FAILURE);
    } else if (retval == 0) {
        printf("Тайм-аут: данные не поступили в течение 5 секунд.\n");
    } else {
        // Проверяем, действительно ли stdin готов к чтению
        if (FD_ISSET(STDIN_FILENO, &readfds)) {
            char buf[100];
            ssize_t n = read(STDIN_FILENO, buf, sizeof(buf) - 1);
            if (n > 0) {
                buf[n] = '\0';
                printf("Прочитано: %s", buf);
            }
        }
    }
    return 0;
}
```

**Пояснения:**

- `select` модифицирует переданные наборы (`fd_set`), поэтому перед каждым новым вызовом их необходимо перенастраивать.
    
- Передаётся максимальный номер дескриптора + 1 (здесь `STDIN_FILENO + 1`), чтобы ядро знало, до какого бита сканировать.
    
- После возврата проверяем готовность с помощью `FD_ISSET`.

### 2. Пример использования `poll`

```c
#include <stdio.h>
#include <stdlib.h>
#include <poll.h>
#include <unistd.h>
#include <string.h>

int main() {
    struct pollfd fds[1];
    int retval;
    // Заполняем структуру для stdin
    fds[0].fd = STDIN_FILENO;
    fds[0].events = POLLIN;        // Ждём событий чтения
    fds[0].revents = 0;             // Будет заполнено ядром
    // Тайм-аут: 5000 миллисекунд = 5 секунд
    retval = poll(fds, 1, 5000);
    if (retval == -1) {
        perror("poll()");
        exit(EXIT_FAILURE);
    } else if (retval == 0) {
        printf("Тайм-аут: данные не поступили в течение 5 секунд.\n");
    } else {
        // Проверяем поле revents
        if (fds[0].revents & POLLIN) {
            char buf[100];
            ssize_t n = read(fds[0].fd, buf, sizeof(buf) - 1);
            if (n > 0) {
                buf[n] = '\0';
                printf("Прочитано: %s", buf);
            }
        }
        // Можно также проверить на ошибки, например POLLERR, POLLHUP
        if (fds[0].revents & (POLLERR | POLLHUP)) {
            fprintf(stderr, "Ошибка или закрытие дескриптора\n");
        }
    }
    return 0;
}
```

**Пояснения:**

- `poll` работает с массивом структур `struct pollfd`. Поле `events` устанавливается приложением, поле `revents` заполняется ядром.
    
- При каждом вызове `poll` ядро копирует весь массив в пространство ядра и сканирует его.
    
- Тайм-аут задаётся в миллисекундах.

### 3. Пример использования `epoll`

```c
#include <stdio.h>
#include <stdlib.h>
#include <sys/epoll.h>
#include <unistd.h>
#include <string.h>

int main() {
    int epfd;
    struct epoll_event ev, events[1];
    int retval;
    // Создаём экземпляр epoll
    epfd = epoll_create1(0);   // флаг 0 означает отсутствие специальных опций
    if (epfd == -1) {
        perror("epoll_create1");
        exit(EXIT_FAILURE);
    }
    // Настраиваем событие для stdin
    ev.events = EPOLLIN;        // Ждём чтения
    ev.data.fd = STDIN_FILENO;  // Сохраняем дескриптор в данных (можно хранить указатель на свою структуру)
    // Регистрируем дескриптор в epoll
    if (epoll_ctl(epfd, EPOLL_CTL_ADD, STDIN_FILENO, &ev) == -1) {
        perror("epoll_ctl");
        close(epfd);
        exit(EXIT_FAILURE);
    }
    // Ожидаем события. Тайм-аут 5000 мс.
    retval = epoll_wait(epfd, events, 1, 5000);
    if (retval == -1) {
        perror("epoll_wait");
        close(epfd);
        exit(EXIT_FAILURE);
    } else if (retval == 0) {
        printf("Тайм-аут: данные не поступили в течение 5 секунд.\n");
    } else {
        // Проверяем события для каждого готового дескриптора
        for (int i = 0; i < retval; i++) {
            if (events[i].events & EPOLLIN) {
                int fd = events[i].data.fd;
                char buf[100];
                ssize_t n = read(fd, buf, sizeof(buf) - 1);
                if (n > 0) {
                    buf[n] = '\0';
                    printf("Прочитано с fd %d: %s", fd, buf);
                }
            }
        }
    }
    close(epfd);
    return 0;
}
```

**Пояснения:**

- `epoll_create1(0)` создаёт контекст epoll, возвращая файловый дескриптор.
    
- Регистрация интереса выполняется один раз через `epoll_ctl`. В структуре `epoll_event` поле `data` может содержать пользовательские данные (здесь мы сохранили сам дескриптор).
    
- `epoll_wait` возвращает только те дескрипторы, на которых произошли события, и копирует их в массив `events` в пространство пользователя.
    
- В примере используется режим **Level-Triggered** (по умолчанию). Если бы мы хотели использовать Edge-Triggered, нужно было бы добавить флаг `EPOLLET` в `ev.events` и перевести дескриптор в неблокирующий режим, а затем в цикле читать до `EAGAIN`.

### Сравнение подходов в примерах

|Механизм|Ключевая особенность в примере|
|---|---|
|`select`|Перед каждым вызовом необходимо заново собирать битовые маски.|
|`poll`|Используется массив структур, поле `revents` разделяет запрошенные и возвращённые события.|

## 1. Историческая справка: Синхронная модель

В ранних версиях UNIX и первых реализациях Linux операции чтения и записи выполнялись **синхронно** и напрямую взаимодействовали с устройством хранения (ПЗУ, например, жесткий диск).

- **Механизм:** Данные передавались непосредственно между буфером в пользовательской памяти (ОЗУ User space) и устройством.
    
- **Следствия:**
    
    1. **Блокировки:** Системный вызов (например, `write`) не возвращал управление приложению до тех пор, пока данные физически не записывались на диск.
        
    2. **Предсказуемость:** После успешного завершения вызова программист мог быть на 100% уверен, что данные находятся на носителе.
        
    3. **Производительность:** Скорость работы программы была жестко ограничена скоростью работы жесткого диска (самого медленного звена), что приводило к простоям центрального процессора [](https://intuit.ru/studies/courses/570/426/lecture/9699?page=3#keyword100).
        

## 2. Современная архитектура: Буферизация (ОЗУ-ОЗУ-ПЗУ)

Современные ядра Linux используют **страничный кэш (page cache)** — область оперативной памяти ядра (Kernel space), предназначенную для кэширования данных файлов. Это позволяет отвязать приложение от скорости физического носителя [](https://sql-ex.ru/blogs/?/Kjesh_stranic_Linux_i_PostgreSQL.html).

![[Pasted image 20260315111849.png|642]]

**Путь записи: User space -> Kernel space -> ПЗУ**

1. Приложение вызывает `write()`. Данные копируются из буфера в пространстве пользователя в буферы ядра (страничный кэш).
    
2. Вызов `write()` немедленно возвращает управление приложению.
    
3. Данные в кэше помечаются как **«грязные» (dirty pages)** — они изменились в памяти, но ещё не синхронизированы с диском.
    
4. Специализированный поток ядра (в современных системах — `flush` или `pdflush` / `kworker`'ы) асинхронно, в фоновом режиме, записывает эти страницы на диск, когда это становится эффективно [](https://habr.com/ru/companies/piter/articles/436986/)[](https://sql-ex.ru/blogs/?/Kjesh_stranic_Linux_i_PostgreSQL.html).
    

**Путь чтения: ПЗУ -> Kernel space -> User space**

1. При первом чтении небольшого объема данных (например, 100 байт) система сталкивается с промахом кэша (cache miss) и читает данные с диска.
    
2. Применяется механизм **упреждающего чтения (read-ahead)**. Ядро, предполагая последовательный доступ, считывает с диска значительно больше данных (например, следующую порцию в 100 байт) в страничный кэш.
    
3. Данные из кэша копируются в пользовательский буфер.
    
4. Последующие операции чтения, если они обращаются к уже закэшированным данным, будут выполняться со скоростью оперативной памяти (cache hit) [](https://sql-ex.ru/blogs/?/Kjesh_stranic_Linux_i_PostgreSQL.html).
    

## 3. Буферизация потоков ввода-вывода (stdio)

Путаница, описанная в запросе с функцией `printf`, относится к ещё одному уровню буферизации — **буферизации пользовательского пространства**, реализованной в библиотеке стандартного ввода-вывода (stdio.h).

- Библиотека `stdio` собирает данные в буфер внутри самого приложения.
    
- Когда буфер заполняется или приложение вызывает `fflush()` (явно или неявно при закрытии), библиотека вызывает системный вызов `write()`.
    
- Данные из буфера `printf` попадают в буферы ядра (page cache), и только оттуда — на экран (или в файл).
    
- **Пример с `sleep()`:** `printf` копирует строку во внутренний буфер `stdout`. Затем вызывается `sleep`. Так как буфер `stdio` не переполнен и принудительно не сброшен, строка не передается ядру. После завершения `sleep` и выхода из `main` срабатывает эпилог кода C (или деструкторы), который вызывает `fflush(stdout)`, и строка наконец попадает в терминал [](https://new2.intuit.ru/studies/professional_skill_improvements/2058/courses/496/lecture/11252).
    

## 4. Когерентность кэша (Coherency)

Механизм буферов гарантирует согласованность данных при чтении и записи одного файла:

1. Если программа записывает данные в буфер записи (грязные страницы), а затем пытается прочитать данные из того же файла до того, как они сброшены на диск, подсистема ввода-вывода проверяет наличие этих данных в кэше страниц.
    
2. Данные будут скопированы непосредственно из буфера записи (грязных страниц) в буфер чтения, минуя физическое устройство, и переданы пользователю. Это гарантирует, что программа всегда увидит свои последние изменения [](https://sql-ex.ru/blogs/?/Kjesh_stranic_Linux_i_PostgreSQL.html).
    

## 5. Основной программный интерфейс (API)

Работа с файлами в Linux начинается с получения файлового дескриптора (file descriptor, FD) — целочисленного индекса в таблице файловых дескрипторов процесса.

**Заголовочные файлы и типы**

```c unfold
#include <fcntl.h>   // Для open, O_RDONLY, O_WRONLY...
#include <unistd.h>  // Для read, write, close, lseek, sync...
#include <sys/uio.h> // Для readv, writev
#include <sys/stat.h> // Для stat
// Специальный тип для подсчета байтов, возвращаемый read/write
// ssize_t — знаковый аналог size_t, позволяющий вернуть -1 в случае ошибки.
```

**Основные системные вызовы**

|Системный вызов|Описание|Примечания|
|---|---|---|
|`int open(const char *path, int flags, mode_t mode)`|Создает файловый дескриптор. Возвращает FD (≥0) или -1 при ошибке.|`flags`: `O_RDONLY`, `O_WRONLY`, `O_RDWR`, побитово с `O_CREAT`, `O_SYNC` и др. `mode`: права доступа при создании файла (umask применяется позже).|
|`int close(int fd)`|Закрывает файловый дескриптор. При закрытии автоматически сбрасываются буферы, относящиеся к этому файловому дескриптору в пользовательском пространстве (не всегда в ядре).|Ядро может отложить физическую запись.|
|`ssize_t read(int fd, void *buf, size_t count)`|Пытается считать до `count` байт из файла `fd` в буфер `buf`.|**Важно:** Возвращает реально считанное количество байт. Может быть меньше запрошенного (конец файла).|
|`ssize_t write(int fd, const void *buf, size_t count)`|Пытается записать `count` байт из буфера `buf` в файл `fd`.|По умолчанию возвращает управление после копирования данных в страничный кэш, а не на диск [](https://intuit.ru/studies/courses/570/426/lecture/9699?page=3#keyword100).|
|`off_t lseek(int fd, off_t offset, int whence)`|Перемещает указатель позиции в файле (file offset) для дескриптора `fd`.|`whence`: `SEEK_SET`, `SEEK_CUR`, `SEEK_END`.|

## 6. Управление синхронизацией (Flushing)

Приложения, критичные к сохранности данных (базы данных, журналирующие системы), должны принудительно гарантировать запись данных на диск.

- **`void sync(void)`**: Запланирует запись всех буферов файловой системы на диск. В современных Linux возвращает управление после того, как данные хотя бы запланированы, но не обязательно записаны [](https://habr.com/ru/companies/piter/articles/436986/).
    
- **`int fsync(int fd)`**: Сбрасывает на диск **и данные, и метаданные** (владелец, права, время изменения), связанные с конкретным файлом `fd`. Возвращает управление только после подтверждения устройства [](https://habr.com/ru/companies/piter/articles/436986/).
    
- **`int fdatasync(int fd)`**: То же, что `fsync`, но сбрасывает на диск **только данные**, необходимые для последующего чтения (целостность данных). Метаданные (например, временные метки `st_atime`/`st_mtime`) могут быть не сброшены, что повышает производительность [](https://habr.com/ru/companies/piter/articles/436986/).
    
- **Флаг `O_SYNC`**: Передается в `open`. Принуждает каждый вызов `write` вести себя как синхронный (ждет физической записи), что резко снижает производительность [](https://intuit.ru/studies/courses/570/426/lecture/9699?page=3#keyword100)[](https://habr.com/ru/companies/piter/articles/436986/).
    

## 7. Разряженный ввод-вывод (Scatter-Gather IO)

Системные вызовы `readv` и `writev` позволяют за один вызов считать данные в несколько не смежных в памяти буферов (разнесенный ввод) или записать данные из нескольких буферов в файл (сборный вывод). Это уменьшает количество системных вызовов и гарантирует атомарность операции для обычных файлов [](https://manpages.ubuntu.com/manpages/noble/ru/man2/writev.2.html).

```c unfold
#include <sys/uio.h>
struct iovec {
    void  *iov_base; // Адрес буфера
    size_t iov_len;  // Размер буфера
};
ssize_t readv(int fd, const struct iovec *iov, int iovcnt);
ssize_t writev(int fd, const struct iovec *iov, int iovcnt);
```
## 8. Ограничения по размеру и архитектурные константы

- **Максимальный размер для `read`/`write`**: Хотя тип `size_t` позволяет запросить передачу гигантского объема данных, ядро Linux ограничивает одну операцию значением `MAX_RW_COUNT`, определенным как `INT_MAX & PAGE_CACHE_MASK` (обычно 0x7FFFF000 или 2,147,479,552 байта для 4K страниц). Это сделано для предотвращения переполнения счетчиков в драйверах и внутренних структурах [](https://git.kernel.org/pub/scm/linux/kernel/git/torvalds/linux.git/commit/?id=435f49a518c78eec8e2edbbadd912737246cbe20).
    
- **`SSIZE_MAX`**: Максимальное значение, которое гарантированно может быть возвращено через `ssize_t`. На практике, если запросить больше, чем позволяет `MAX_RW_COUNT`, ядро обрежет запрос до этого лимита.
    
- **`IOV_MAX` / `UIO_MAXIOV`**: Лимит на количество векторов (`iovcnt`) в вызовах `readv`/`writev`. В современных Linux — 1024 [](https://manpages.ubuntu.com/manpages/noble/ru/man2/writev.2.html).
    

## 9. Современная эволюция: `io_uring`

Начиная с ядра 5.1, в Linux появился новый интерфейс асинхронного ввода-вывода — **`io_uring`**. Он использует пару кольцевых буферов, отображаемых в память пользовательского пространства, что позволяет минимизировать количество копирований и системных вызовов. Это эволюция концепции буферизации, где приложение и ядро эффективно обмениваются пакетами запросов ввода-вывода (Submission Queue и Completion Queue) [](https://niceos.ru/site/package-view?id=1143). Библиотека `liburing` предоставляет удобную обертку над этим механизмом.

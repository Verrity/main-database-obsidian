### **1. Потоки в Linux: фундаментальные концепции**

В операционных системах семейства Linux поток (thread) реализуется как _легковесный процесс_ (Lightweight Process) и является диспетчеризируемой единицей (unit of dispatch). Поток не может существовать вне процесса — он всегда выполняется в контексте конкретного процесса, разделяя с ним большинство ресурсов.

- **Определение:** Поток представляет собой последовательность инструкций (поток команд), исполняемых в контексте процесса. Потоки часто ассоциируют с функциями, которые они выполняют.
    
- **Диспетчеризация:** Планировщик ядра оперирует именно потоками. С точки зрения ядра Linux, процесс и поток — это задачи (`task_struct`), но флаги, переданные при создании через системный вызов `clone()`, определяют, какие ресурсы будут разделяться с родителем.
    
- **Накладные расходы:** Создание потока и переключение контекста между потоками требуют значительных затрат процессорного времени (сохранение/восстановление регистров, обновление структур планировщика, потенциальная инвалидация TLB). Поэтому для небольших объёмов вычислений (например, 100 млн инкрементов) последовательный код выполняется быстрее многопоточного. Параллелизм начинает оправдывать себя при очень больших вычислительных нагрузках (100 млрд операций), когда выигрыш от распараллеливания превышает издержки.
    

### **2. Сравнение процессов и потоков**

- **Процессы:** Изолированы друг от друга — не могут напрямую обращаться к адресному пространству другого процесса.
    
- **Потоки:** Разделяют адресное пространство одного процесса, но не разделяют данные друг друга напрямую (доступ к разделяемым данным требует синхронизации).

Процессы справа - потоки слева
![[Pasted image 20260315160734.png|747]]

**Разделяемые ресурсы потока с процессом:**

- Адресное пространство (сегменты кода, данных, хип).
    
- Идентификаторы процесса (PID, группа процессов, сессия).
    
- Таблица файловых дескрипторов.
    
- Обработчики сигналов (но маска сигналов у каждого потока своя).
    
- Информация о файловой системе (текущий рабочий каталог, корневой каталог).
    

**Уникальные атрибуты потока:**

- Идентификатор потока (TID, возвращаемый `pthread_self()`).
    
- Счётчик инструкций (Program Counter) — каждый поток выполняет свою последовательность команд.
    
- Набор регистров процессора, включая указатель стека (SP).
    
- Собственный стек.
    
- Копия переменной `errno` (обычно реализована через thread-local storage).
    
- Маска сигналов (может быть изменена).
    
- Приоритет и политика планирования (наследуются от процесса, но могут быть переопределены).
    
- Специфичные данные потока (Thread-Specific Data) — уникальное хранилище, доступное только этому потоку через специальный API.
    

**Важно:** Вне процесса создать поток невозможно.

### **3. Жизненный цикл и взаимодействие с процессами**

![[Pasted image 20260315160803.png|413]]

#### **3.1. Поведение при вызове `fork()` в многопоточной программе**

Внутри потока можно вызвать `fork()`, но результат отличается от однопоточной программы:

- Вызов `fork()` дублирует только вызвавший поток. В дочернем процессе будет существовать лишь один поток — тот, который вызвал `fork()`.
    
- Дочерний процесс начинает выполнение с точки возврата из `fork()` в том же контексте функции, где был сделан вызов. Функция `main()` повторно не выполняется.
    
- Если в процессе G существуют потоки A, B, C, и поток B вызывает `fork()`, то в новом процессе L будет только один поток (копия B), который продолжит выполнение после `fork()`.
    

#### **3.2. Завершение потоков и процессов**

- **Завершение потока:** Для корректного завершения отдельного потока используется функция `pthread_exit()`. Вызов стандартной функции `exit()` завершает **весь процесс** и все его потоки, что в многопоточной программе обычно нежелательно.
    
- **Главный поток:** Внутри библиотеки `libc` существует главный поток, выполняющий функцию `main()`. Если главный поток завершается (возвратом из `main` или вызовом `exit()`), процесс завершается.
    
- **Отсоединённые потоки (detached):** Поток может быть создан в отсоединённом состоянии или переведён в него позже (`pthread_detach()`). Такой поток автоматически освобождает свои ресурсы при завершении; к нему нельзя применить `pthread_join()`, и он никогда не становится зомби.
    
- **Убийство процесса:** Если процесс уничтожен (например, сигналом SIGKILL), все его потоки завершаются принудительно. Необходимо дождаться завершения каждого потока через `pthread_join()` (если он не отсоединён) для корректной очистки.
    

### **4. Проблемы параллелизма: гонки, синхронизация и свойства функций**

#### **4.1. Гонка данных (Data Race)**

Гонка возникает, когда два или более потока одновременно обращаются к одной разделяемой переменной, и хотя бы одно обращение является записью. Результат выполнения недетерминирован и часто приводит к некорректным данным.

Пример: один поток пишет, другой читает одновременно — может быть считана только часть обновлённого значения.

**Следствие:** Для борьбы с гонкой применяются примитивы синхронизации (мьютексы, семафоры и т.д.), но их использование приводит к сериализации выполнения критических секций, что снижает параллелизм.

#### **4.2. Сериализация (в контексте потоков)**

В языке C термин «сериализация» обычно означает преобразование объекта в последовательность байт. Однако в контексте многопоточности под **сериализацией** понимается **последовательное выполнение потоков** — когда потоки вынуждены ждать друг друга для доступа к разделяемым ресурсам.

#### **4.3. Реентерабельность (Reentrancy)**

Реентерабельная функция гарантирует корректную работу даже в том случае, если её выполнение прервано (например, сигналом или другим потоком) и функция вызвана снова до завершения первого вызова. Реентерабельность подразумевает **отсутствие гонок и возможность параллельного выполнения без блокировок**. Обычно такие функции не используют статические или глобальные данные, а работают только с локальными переменными и аргументами.

#### **4.4. Потокобезопасность (Thread Safety)**

Функция является потокобезопасной, если она может быть одновременно вызвана из нескольких потоков и при этом продолжает корректно работать с разделяемыми данными.

- Потокобезопасность гарантирует **отсутствие конкуренции за данные** (благодаря внутренней синхронизации или отсутствию разделяемых данных).
    
- **Потокобезопасность не гарантирует параллельности!** Использование блокировок внутри функции может сериализовать все вызовы, сводя на нет преимущества многопоточности.
    

### **5. API создания и управления потоками (POSIX Threads)**

Для компиляции программ с pthreads необходимо указывать ключ `-pthread`.

```c unfold
#include <pthread.h>
int pthread_create(pthread_t *thread, const pthread_attr_t *attr,
                   void *(*start_routine)(void *), void *arg);
```

- `thread` — указатель на переменную типа `pthread_t`, куда будет записан идентификатор нового потока.
    
- `attr` — атрибуты потока (можно передать `NULL` для атрибутов по умолчанию). **Примечание:** в некоторых дистрибутивах Linux по умолчанию создаются отсоединённые (detached) потоки, поэтому рекомендуется явно задавать атрибуты.
    
- `start_routine` — указатель на функцию, которую будет выполнять поток. Она принимает `void*` и возвращает `void*`.
    
- `arg` — указатель на аргументы функции.
    
- Возвращаемое значение: `0` при успехе, код ошибки в противном случае. Это неблокирующая функция.
    

```c unfold
int pthread_join(pthread_t thread, void **retval);
```

- Блокирующая функция, ожидающая завершения указанного потока.
    
- `thread` — идентификатор потока.
    
- `retval` — указатель, куда будет помещён возвращаемый потоком статус (значение, переданное в `pthread_exit()` или возвращённое из `start_routine`).
    
- Для корректного завершения необходимо вызвать `pthread_join` для каждого присоединяемого (не detached) потока.
    

```c unfold
void pthread_exit(void *retval);
```

- Завершает вызвавший поток, возвращая `retval` (может быть получено другим потоком через `pthread_join`).
    

```c unfold
int pthread_cancel(pthread_t thread);
```

- Посылает запрос на отмену указанному потоку. Реакция на отмену зависит от состояния отмены потока (настраивается через `pthread_setcancelstate` и `pthread_setcanceltype`).
    

```c unfold
pthread_t pthread_self(void);
```

- Возвращает идентификатор текущего потока.
    

```c unfold
int pthread_equal(pthread_t t1, pthread_t t2);
```

- Макрос (или функция) для сравнения двух идентификаторов потоков.
    

```c unfold
int pthread_setcancelstate(int state, int *oldstate);
int pthread_setcanceltype(int type, int *oldtype);
```

- Управляют поведением потока при получении запроса на отмену: можно разрешить/запретить отмену, а также выбрать тип завершения — немедленно (`PTHREAD_CANCEL_ASYNCHRONOUS`) или только в точках отмены (`PTHREAD_CANCEL_DEFERRED`).
    

### **6. Практические примеры и разбор типичных ошибок**

#### **6.1. Пример 1: Передача индекса потоку (проблема с указателем на счётчик цикла)**

```c
#include <stdio.h>
#include <pthread.h>
#define N 5
long a = 0;
pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;
void* thread_calc(void* args){
    int* i = (int*)args;
    printf("thread num %d\n", *i);
    return NULL;
}
int main(void)
{
    int i, i1;
    int* s;
    pthread_t thread[N];
    for (i = 0; i < N; i++){
        pthread_create(&thread[i], NULL, thread_calc, (void*)&i);
    }
    for (i = 0; i < N; i++){
        pthread_join(thread[i], (void**)&s);
    }
    return 0;
}
```

**Ожидание:** номера потоков от 0 до 4 вперемешку.  
**Реальность:** дублирование номеров (например, несколько троек, несколько нулей).  

![[Pasted image 20260315161004.png|279]]

**Причина:** В цикле создания передаётся указатель на одну и ту же локальную переменную `i`, которая изменяется на каждой итерации. Потоки могут начать выполнение в любой момент после создания. К моменту, когда поток читает значение по указателю, переменная `i` уже могла быть увеличена или даже выйти за пределы цикла.

**Анализ вывода:**

- Две тройки — два потока успели запуститься, пока счётчик достиг третьей итерации.
    
- Три нуля — оставшиеся три потока запустились, когда основной поток уже завершил цикл создания и перешёл к ожиданию (на нулевой итерации цикла ожидания).
    

**Неправильное исправление:** замена `i` на `i1` в цикле `pthread_join` не решает проблему, так как передача адреса всё ещё остаётся:

```c unfold
for (i1 = 0; i1 < N; i1++){
    pthread_join(thread[i1], (void**)&s);
}
```

![[Pasted image 20260315161138.png|271]]

Дублирование сохраняется, потому что зависимость от изменяемой переменной остаётся.

**Правильное решение:** создать массив для хранения копий значений индексов и передавать адрес элемента этого массива.

```c
int main(void)
{
    int i, i1;
    int* s;
    pthread_t thread[N];
    int v[N];  // массив для индексов
    for (i = 0; i < N; i++){
        v[i] = i;
        pthread_create(&thread[i], NULL, thread_calc, (void*)&v[i]);
    }
    for (i1 = 0; i1 < N; i1++){
        pthread_join(thread[i1], (void**)&s);
    }
    return 0;
}
```

![[Pasted image 20260315161200.png|227]]
Теперь каждый поток получает указатель на уникальный элемент массива, значение которого не меняется.

#### **6.2. Пример 2: Гонка при инкременте общей переменной**

**Последовательная версия** (180 млн инкрементов) выполняется за 0.3 секунды и даёт корректный результат.

```c
#include <stdio.h>
int main(void){
    int i;
    long a = 0;
    for (i = 0; i < 180000000; i++)
        a++;
    printf("a = %ld\n", a);
    return 0;
}
```

![[Pasted image 20260315161220.png]]

**Попытка распараллелить:** 18 потоков, каждый инкрементирует общую переменную `a` 10 млн раз. Ожидание: ускорение в 18 раз и результат 180 млн.

```c
#include <stdio.h>
#include <pthread.h>
#define N 18
long a = 0;
pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;
void* thread_calc(void* args)
{
    int i;
    for (i = 0; i < 10000000; i++){
        a++;
    }
    return NULL;
}
int main(void)
{
    int i, i1;
    int* s;
    pthread_t thread[N];
    for (i = 0; i < N; i++){
        pthread_create(&thread[i], NULL, thread_calc, NULL);
    }
    for (i1 = 0; i1 < N; i1++){
        pthread_join(thread[i1], (void**)&s);
    }
    
    printf("a = %ld\n", a);
    return 0;
}
```

![[Pasted image 20260315161240.png|474]]
**Результат:** загрузка 17 ядер, время выполнения значительно больше, итоговое значение `a` = около 13 млн вместо 180 млн.

**Причина гонки:** Операция `a++` не атомарна — она состоит из чтения, модификации и записи. Все 18 потоков одновременно читают одно и то же значение `a` (например, 1000), каждый увеличивает его до 1001 и записывает обратно. В итоге вместо 18 прибавлений выполняется только одно, остальные 17 перезаписывают результат друг друга.

Для устранения гонки необходимы примитивы синхронизации, например мьютексы (рассматриваются в разделе «Примитивы потоков»).

### **7. Потокобезопасность и специфичные данные потока (Thread-Specific Data)**

#### **7.1. Пример небезопасной функции с глобальным буфером**

```c
c

#include <stdio.h>
#include <string.h>
#include <malloc.h>
char* buff;  // глобальный буфер
char* error_str(int no)
{
    switch(no) {
        case 1: strcpy(buff, "Error 1"); break;
        case 2: strcpy(buff, "Error 2"); break;
    }
    return buff;
}
```

Функция не является потокобезопасной: два потока могут одновременно писать в один и тот же буфер, перетирая данные друг друга.

#### **7.2. Исправление с помощью мьютекса**

```c
char* error_str(int no)
{
    pthread_mutex_lock(&m1);
    switch(no) {
        case 1: strcpy(buff, "Error 1"); break;
        case 2: strcpy(buff, "Error 2"); break;
    }
    pthread_mutex_unlock(&m1);
    return buff;
}
```

Мьютекс предотвращает одновременный доступ, но сериализует вызовы. При этом функция всё ещё использует общий буфер.

#### **7.3. Исправление выделением памяти при каждом вызове**

```c
char* error_str(int no)
{
    char* buff = malloc(100);  // malloc потокобезопасен
    switch(no) {
        case 1: strcpy(buff, "Error 1"); break;
        case 2: strcpy(buff, "Error 2"); break;
    }
    return buff;
}
```

Теперь каждый вызов получает свой буфер, но:

- Буфер нужно освобождать снаружи.
    
- Функция всё ещё сериализуется на вызове `malloc` (если `malloc` использует глобальные блокировки), поэтому параллельность не достигается.
    

#### **7.4. Приватные данные потока (Thread-Specific Data)**

Механизм TSD позволяет создать хранилище, уникальное для каждого потока, доступное через глобальный ключ. Это эффективный способ избежать блокировок и повторных выделений памяти.

**Принцип работы:**

- Создаётся глобальный ключ типа `pthread_key_t`.
    
- Для ключа ядро (или библиотека pthreads) поддерживает неявную хеш-таблицу: для каждого потока ключу соответствует своё значение.
    
- При завершении потока можно вызвать деструктор для очистки данных.
    

**Пример реализации потокобезопасной `error_str` с TSD и однократной инициализацией:**

```c
#include <stdio.h>
#include <string.h>
#include <pthread.h>
#include <malloc.h>
pthread_once_t once = PTHREAD_ONCE_INIT;
pthread_key_t key;
// Деструктор, вызываемый при завершении потока
void dest(void* buff){
    free(buff);
}
// Функция однократного создания ключа
void key_create(void) {
    pthread_key_create(&key, dest);
}
char* error_str(int no)
{
    char* buff;
    // pthread_once гарантирует, что key_create выполнится только один раз во всём процессе
    pthread_once(&once, key_create);
    // Получаем значение, специфичное для текущего потока
    buff = pthread_getspecific(key);
    if (buff == NULL) {
        // Поток ещё не имеет своего буфера — создаём
        buff = malloc(255);
        pthread_setspecific(key, buff);
    }
    switch(no) {
        case 1: strcpy(buff, "Error 1"); break;
        case 2: strcpy(buff, "Error 2"); break;
    }
    return buff;
}

```

**Пояснения:**

- `pthread_once` гарантирует, что ключ будет создан ровно один раз, независимо от количества потоков, вызывающих `error_str`.
    
- Каждый поток при первом входе выделяет свой буфер и сохраняет его по ключу. Последующие вызовы в том же потоке используют уже существующий буфер.
    
- При завершении потока деструктор `dest` автоматически освободит буфер.
    

Этот подход не требует блокировок в критическом пути (кроме однократной инициализации) и позволяет потокам работать параллельно.

### **8. Дополнительные замечания**

- **Крупноблочное распараллеливание:** Когда потоки работают над непересекающимися областями памяти одной задачи (например, обработка разных кусков массива), синхронизация не требуется, и параллелизм достигается эффективно. Однако большинство задач не удаётся так просто разбить.
    
- **Порядок создания потоков:** Потоки создаются не в той последовательности, в которой вызывается `pthread_create`; реальный порядок запуска определяется планировщиком и непредсказуем.
    
- **Detached-потоки:** В некоторых дистрибутивах Linux потоки по умолчанию могут создаваться отсоединёнными. Это важно учитывать, иначе вызов `pthread_join` для такого потока приведёт к ошибке.

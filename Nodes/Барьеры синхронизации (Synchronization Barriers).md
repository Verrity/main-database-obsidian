Это классический примитив синхронизации, также известный как "точка рандеву" (rendezvous point). Он заставляет группу потоков (или процессов) ожидать в заданной точке кода до тех пор, пока все участники не достигнут её.

#### 1. Реализация в ядре Linux

В ядре существует реализация масштабируемого барьера (scalable synchronization barrier), предложенная Питером Зейлстрой (Peter Zijlstra). Она спроектирована так, чтобы минимизировать инвалидацию кэш-линий (cacheline bouncing) [](https://lkml.indiana.edu/0701.3/1243.html).

Структура барьера (`struct barrier`) содержит [](https://lkml.indiana.edu/0701.3/1243.html):

- `atomic_t count`: Атомарный счетчик активных секций блокировки.
    
- `wait_queue_head_t wait`: Очередь ожидания для потоков, вызвавших синхронизацию.
    

#### 2. Принцип работы

- **`barrier_lock()` / `barrier_unlock()`:** Эти функции обрамляют критические секции, которые являются "редкими" по сравнению с операциями синхронизации. `barrier_lock` увеличивает атомарный счетчик и использует `smp_wmb()` для упорядочивания. `barrier_unlock` уменьшает счетчик и, если он становится равен нулю, пробуждает все потоки, ожидающие на очереди `wait` [](https://lkml.indiana.edu/0701.3/1243.html).
    
- **`barrier_sync()`:** Это "горячий" (частый) путь. Функция проверяет значение атомарного счетчика. Если счетчик равен нулю (нет активных lock-секций), она завершается немедленно, не выполняя дорогостоящих операций. Если счетчик не ноль, поток переводится в состояние ожидания (`TASK_UNINTERRUPTIBLE`) в очереди `wait` до тех пор, пока все lock-секции не будут завершены [](https://lkml.indiana.edu/0701.3/1243.html).
    

```c unfold
static inline void barrier_sync(struct barrier *b)
{
    might_sleep();
    if (unlikely(atomic_read(&b->count))) {
        // ... подготовка к ожиданию
        while (atomic_read(&b->count))
            schedule(); // уход в сон до пробуждения в barrier_unlock
        // ... завершение ожидания
    }
}
```

Такой подход гарантирует, что в отсутствие конфликтов (когда никто не удерживает lock) вызов `barrier_sync` выполняется предельно быстро — это всего лишь одна атомарная операция чтения.

#### 3. Роль фьютексов (Futexes)

Фьютексы (Fast Userspace Mutex) являются низкоуровневым строительным блоком для множества примитивов синхронизации в Linux. Они позволяют реализовать ожидание и пробуждение потоков, эффективно работая в пользовательском пространстве в бесконфликтных ситуациях и обращаясь к ядру только при реальной необходимости. Они могут использоваться в том числе и для организации барьеров, позволяя одновременно пробуждать группу задач [](https://ru.wikipedia.org/wiki/%D0%A4%D1%8C%D1%8E%D1%82%D0%B5%D0%BA%D1%81).

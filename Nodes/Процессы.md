### **Определение и Представление в Ядре**

С точки зрения ядра Linux, процесс — это не просто выполняющаяся программа, а абстракция, предоставляющая среду для выполнения машинного кода. В контексте ядра процесс (или «задача», англ. task) — это экземпляр исполняемого файла, которому сопоставлены уникальные атрибуты и ресурсы.

Основной структурой данных, описывающей процесс в ядре, является `struct task_struct` (часто называемая `task_struct`). В более новых версиях ядра название сохраняется, но структура обрастает новыми полями. Каждый процесс в системе представлен своим экземпляром `task_struct`, который содержит всю информацию о нём: идентификаторы, состояния, права доступа, счётчики использования ресурсов, указатели на области виртуальной памяти (text, data, stack), таблицу файловых дескрипторов, контекст сигналов и многое другое [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12). Все `task_struct` связаны в двусвязный список, что позволяет ядру итерировать по всем процессам, например, с помощью макроса `for_each_task()` [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12). Для быстрого поиска процесса по его идентификатору (PID) используется хеш-таблица `pidhash` [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12).

Карту виртуальной памяти конкретного процесса можно посмотреть через псевдо-файловую систему `/proc` (например, `cat /proc/<PID>/maps`), которая читает данные из соответствующих полей `task_struct` [](https://javarush.com/quests/lectures/ru.javarush.system.linux.lecture.level02.lecture05?post=full#discussion)[](https://www.zabbix.com/documentation/7.0/ru/manual/appendix/items/proc_get).

### **Атрибуты и Идентификаторы Процесса**

![[Pasted image 20260315124929.png|683]]

Каждый процесс обладает набором идентификаторов, определяющих его место в иерархии, права и принадлежность к группам.

- **PID (Process ID)** — уникальный идентификатор процесса. Ядро назначает его, выбирая первый свободный номер (число ограничено размером типа `pid_t`, но может быть изменено через `/proc/sys/kernel/pid_max`) [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12). Процесс с PID 0 — это процесс ядра (`swapper`), а самый первый пользовательский процесс (`init` или `systemd`) всегда имеет PID 1 [](https://javarush.com/quests/lectures/ru.javarush.system.linux.lecture.level02.lecture05?post=full#discussion).
    
- **PPID (Parent Process ID)** — идентификатор родительского процесса. Все процессы в Linux выстроены в единую иерархию, корнем которой является процесс `init` (PID 1). Это необходимо для корректного управления сигналами и завершения работы системы [](https://raw.githubusercontent.com/0voice/linux_kernel_wiki/refs/heads/main/%E6%96%87%E7%AB%A0/%E8%BF%9B%E7%A8%8B%E7%AE%A1%E7%90%86/%E8%BF%9B%E7%A8%8B%E7%9A%84%E6%A0%B8%E5%BF%83%E2%80%94%E2%80%94task_truct.md). Поле `parent` в `task_struct` указывает на родителя, а `children` — на список дочерних процессов [](https://raw.githubusercontent.com/0voice/linux_kernel_wiki/refs/heads/main/%E6%96%87%E7%AB%A0/%E8%BF%9B%E7%A8%8B%E7%AE%A1%E7%90%86/%E8%BF%9B%E7%A8%8B%E7%9A%84%E6%A0%B8%E5%BF%83%E2%80%94%E2%80%94task_truct.md).
    
- **GID (Group ID)** — идентификатор группы процессов. Группы используются для управления набором связанных процессов, например, созданных при работе конвейера shell (`ls | grep test`). Сигнал, посланный группе, будет доставлен всем процессам в ней. Одна группа может принадлежать только одной сессии [](https://www.ccfit.nsu.ru/~deviv/courses/unix/unix/ng4ff02.html).
    
- **SID (Session ID)** — идентификатор сессии. Сессия объединяет одну или несколько групп процессов и привязана к терминалу. Обычно SID равен PID процесса, создавшего сессию (лидера сессии). Сессии необходимы для изоляции работы разных пользователей и управления фоновыми задачами. Если лидер сессии (например, терминал) завершается, ядро может отправить сигнал SIGHUP всем процессам в этой сессии, что приводит к их завершению [](https://masters.donntu.ru/2009/fvti/ivanovy/library/tez1.htm).
    

#### **Идентификаторы пользователя (UID) и механизм Capabilities**

Права процесса определяются набором идентификаторов пользователя, хранящихся в структуре `cred` внутри `task_struct` [](https://raw.githubusercontent.com/0voice/linux_kernel_wiki/refs/heads/main/%E6%96%87%E7%AB%A0/%E8%BF%9B%E7%A8%8B%E7%AE%A1%E7%90%86/%E8%BF%9B%E7%A8%8B%E7%9A%84%E6%A0%B8%E5%BF%83%E2%80%94%E2%80%94task_truct.md):

- **`uid` (Real UID)** — реальный идентификатор пользователя, запустившего процесс.
    
- **`euid` (Effective UID)** — эффективный идентификатор, используемый для большинства проверок прав (доступ к ресурсам, сигналам). Именно он определяет, от чьего имени процесс выполняется в данный момент.
    
- **`suid` (Saved UID)** — сохранённый идентификатор, позволяющий переключаться между реальным и эффективным UID.
    
- **`fsuid` (Filesystem UID)** — идентификатор для проверки прав доступа к файловой системе, обычно совпадает с `euid`.
    

Кроме классической модели UID, Linux использует механизм **Capabilities**. Вместо бинарной привилегии root/не-root, процессу могут быть выданы отдельные "права" (capabilities), такие как `CAP_NET_BIND_SERVICE` (возможность открывать порты <1024) или `CAP_KILL` (возможность посылать сигналы чужим процессам). Это повышает безопасность, так как процессу можно дать ровно столько прав, сколько необходимо [](https://raw.githubusercontent.com/0voice/linux_kernel_wiki/refs/heads/main/%E6%96%87%E7%AB%A0/%E8%BF%9B%E7%A8%8B%E7%AE%A1%E7%90%86/%E8%BF%9B%E7%A8%8B%E7%9A%84%E6%A0%B8%E5%BF%83%E2%80%94%E2%80%94task_truct.md).

### **Состояния Процесса**

#### **Пользовательская модель состояний**

![[Pasted image 20260315125024.png|567]]

В упрощённой модели, процесс может находиться в одном из трёх основных состояний:

1. **Действие (Running)** — выполняется на процессоре в данный момент.
    
2. **Готовность (Runnable)** — ждёт, когда планировщик выделит ему процессорное время.
    
3. **Блокировка (Blocked/Sleeping)** — ждёт наступления какого-либо события (например, ввода данных с диска или по сети).
    

#### **Модель состояний ядра**

![[Pasted image 20260315125539.png]]

В ядре состояния описываются более детально и хранятся в поле `state` структуры `task_struct` [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12)[](https://raw.githubusercontent.com/0voice/linux_kernel_wiki/refs/heads/main/%E6%96%87%E7%AB%A0/%E8%BF%9B%E7%A8%8B%E7%AE%A1%E7%90%86/%E8%BF%9B%E7%A8%8B%E7%9A%84%E6%A0%B8%E5%BF%83%E2%80%94%E2%80%94task_truct.md):

- **`TASK_RUNNING`** — процесс либо выполняется на процессоре, либо находится в очереди на выполнение (runqueue). Это состояние объединяет пользовательские "Действие" и "Готовность".
    
- **`TASK_INTERRUPTIBLE`** — процесс находится в состоянии сна (ожидания события). Он может быть разбужен как по наступлению события, так и по получении сигнала. Все пользовательские процессы могут входить в это состояние.
    
- **`TASK_UNINTERRUPTIBLE`** — "глубокий" сон. Процесс ожидает события напрямую (например, завершения операции ввода-вывода) и не реагирует на сигналы. Это состояние критически важно для некоторых операций ядра и может приводить к появлению "неубиваемых" процессов при проблемах с оборудованием. Находиться в нём могут только процессы или потоки ядра.
    
- **`TASK_STOPPED`** — процесс остановлен, обычно по сигналу `SIGSTOP`, `SIGTSTP` и т.п.
    
- **`TASK_TRACED`** — процесс остановлен отладчиком (например, `ptrace`).
    
- **`EXIT_ZOMBIE`** — процесс завершился, но его родитель ещё не запросил код возврата (не вызвал `wait()`). Все ресурсы процесса (память, файловые дескрипторы) освобождены, но сама структура `task_struct` сохраняется в таблице процессов, чтобы родитель мог узнать статус завершения. Такие процессы называются "зомби".
    
- **`EXIT_DEAD`** — финальное состояние после того, как родитель забрал статус, и процесс окончательно удаляется из системы.
    

### **Многозадачность и Планировщик**

![[Pasted image 20260315125149.png|609]]

Linux реализует **вытесняющую многозадачность** (preemptive multitasking) [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12). Это означает, что решение о том, какой процесс и как долго будет выполняться, принимает планировщик (scheduler). Каждому процессу выделяется **квант времени** (time slice). По истечении кванта или при наступлении события более высокого приоритета, планировщик вытесняет текущий процесс и переключает контекст на другой.

- **Контекст переключения (Context Switch)** — операция сохранения состояния одного процесса (регистры, счётчик команд) и загрузки состояния другого.
    
- **Голодание (Starvation)** — ситуация, когда процесс с низким приоритетом не получает процессорного времени из-за постоянной работы более высокоприоритетных процессов.
    
- **Гонка (Race Condition)** — состояние системы, при котором результат работы зависит от того, в каком порядке выполняются процессы или потоки.
    
- **Инверсия приоритета (Priority Inversion)** — проблема, когда высокоприоритетный процесс косвенно блокируется низкоприоритетным, который удерживает необходимый ресурс. Современные планировщики (как в Linux) используют механизмы вроде Priority Inheritance для её решения.
    

### **Создание Процессов: `fork()` и `exec()`**

Создание нового процесса в Linux — двухэтапный процесс, реализованный системными вызовами `fork()` и `exec()` [](https://cloud.tencent.cn/developer/information/linux%E8%BF%9B%E7%A8%8B%E5%88%9B%E5%BB%BA-article)[](https://worktile.com/kb/ask/359688.html).

1. **`fork()`** — создаёт новый процесс, являющийся практически полной копией родительского [](https://worktile.com/kb/ask/359688.html). Ядро:
    
    - Выделяет новую структуру `task_struct`.
        
    - Копирует (или, в современных реалиях, помечает как "копируемые при записи" — **Copy-on-Write, COW**) адресное пространство родителя.
        
    - Копирует таблицу файловых дескрипторов. Это означает, что дочерний процесс наследует все открытые родителем файлы, сокеты и т.д., и они могут работать с ними независимо [](https://cloud.tencent.cn/developer/information/linux%E8%BF%9B%E7%A8%8B%E5%88%9B%E5%BB%BA-article).
        
    - Различает родителя и потомка по возвращаемому значению: в родителе `fork()` возвращает PID потомка, а в потомке — `0`.
        
2. **`exec()`** — семейство функций (`execl`, `execvp`, `execve`), которые заменяют текущий образ процесса новым [](https://cloud.tencent.cn/developer/information/linux%E8%BF%9B%E7%A8%8B%E5%88%9B%E5%BB%BA-article)[](https://worktile.com/kb/ask/359688.html). Ядро загружает указанный бинарный файл в адресное пространство процесса и запускает его с новой точки входа. При этом большинство атрибутов процесса (PID, PPID, файловые дескрипторы, если не выставлен флаг `FD_CLOEXEC`) сохраняются [](https://cloud.tencent.cn/developer/information/linux%E8%BF%9B%E7%A8%8B%E5%88%9B%E5%BB%BA-article).
    
Именно так работают командные оболочки (shell), например `bash`: при вводе команды `ls`, `bash` выполняет `fork()`, а затем в дочернем процессе вызывает `exec()` для загрузки `/bin/ls`.

![[Pasted image 20260315125222.png|478]]
#### **Важные нюансы `fork()`**

- **Буферизация ввода-вывода:** Стандартные библиотеки (libc) используют буферизацию. При вызове `fork()` копируется не только память процесса, но и содержимое пользовательских буферов библиотек. Это приводит к тому, что данные, ещё не сброшенные в ядро (например, при выводе в терминал без символа `\n` или при перенаправлении вывода в файл), могут быть записаны дважды — один раз при сбросе буфера родителя, второй — при сбросе буфера потомка. Для гарантированного сброса используется `fflush(NULL)` перед `fork()`.
    
- **Наследование:** Как сказано в исходном тексте, `exec()` затирает информацию о бинарном файле, но наследует такие вещи, как таблица дескрипторов и окружение (если не указано иное).
    

### **Типы Процессов**

- **Системные процессы (Kernel Threads)** — создаются ядром и выполняются только в пространстве ядра (например, `kthreadd`, `kswapd`).
    
- **Пользовательские процессы** — все процессы, запущенные из пользовательского пространства.
    
- **Демоны (Daemons)** — специальный тип пользовательских процессов, работающих в фоне и не привязанных к терминалу [](https://masters.donntu.ru/2009/fvti/ivanovy/library/tez1.htm). Для создания демона процесс должен:
    
    1. Выполнить `fork()`, а родительский процесс — завершиться (`exit`). Это гарантирует, что демон не будет лидером группы.
        
    2. Вызвать `setsid()`, чтобы создать новую сессию, стать её лидером и оторваться от управляющего терминала.
        
    3. Сменить текущий рабочий каталог на корневой (`chdir("/")`), чтобы не блокировать размонтирование файловой системы, из которой он был запущен.
        
    4. Закрыть все файловые дескрипторы, унаследованные от родителя (особенно стандартные потоки ввода/вывода/ошибок), чтобы не удерживать терминал.
        
    5. Вся эта логика инкапсулирована в функцию `daemon()`.
        

### **Приоритеты и Привязка к Процессору**



Приоритет процесса (niceness, NI) — это значение от **-20** (наивысший приоритет) до **+19** (наинизший) [](https://manpages.debian.org/bookworm-backports/manpages-ru/nice.1.ru.html). Обычный пользователь может только понижать приоритет своих процессов (увеличивать число nice), в то время как суперпользователь (root) может как понижать, так и повышать его. Приоритет наследуется при `fork()`. Поле `prio` в `task_struct` используется планировщиком для принятия решений.

С помощью системного вызова `sched_setaffinity()` (и утилиты `taskset`) можно задать процессу **привязку к процессору (CPU affinity)**, указав битовую маску ядер, на которых данный процесс может выполняться. Это не даёт монопольного владения ядром, но позволяет оптимизировать кэш-локальность и производительность в NUMA-системах.

### **Пример кода с пояснениями**

```c
#include <stdio.h>
#include <unistd.h>
#include <sys/types.h>
#include <sys/wait.h>
#include <stdlib.h>

int main(void) {
    pid_t child_pid;
    int status;
    int local_var = 10;
    printf("Hello! (PID: %d)\n", getpid());
    fflush(NULL); // Сбрасываем буферы libc перед fork, чтобы избежать их дублирования
    child_pid = fork();
    if (child_pid == -1) {
        perror("fork failed");
        exit(EXIT_FAILURE);
    }
    if (child_pid == 0) {
        // Дочерний процесс
        local_var = 20; // Изменяем копию переменной (благодаря COW)
        printf("Child process: PID = %d, PPID = %d, local_var = %d\n",
               getpid(), getppid(), local_var);
        
        // Заменяем образ дочернего процесса на /bin/ls
        execl("/bin/ls", "ls", "-la", NULL);
        
        // Если execl вернулся, значит произошла ошибка
        perror("execl failed");
        exit(EXIT_FAILURE);
    } else {
        // Родительский процесс
        printf("Parent process: PID = %d, child's PID = %d, local_var = %d\n",
               getpid(), child_pid, local_var);
        
        // Ожидаем завершения дочернего процесса
        if (wait(&status) == -1) {
            perror("wait failed");
            exit(EXIT_FAILURE);
        }
        if (WIFEXITED(status)) {
            printf("Child exited with status = %d\n", WEXITSTATUS(status));
        } else if (WIFSIGNALED(status)) {
            printf("Child was killed by signal %d\n", WTERMSIG(status));
        }
    }
    return 0; // exit(0) будет вызван неявно
}
```

**Пояснения:**

- `fflush(NULL)` перед `fork()` сбрасывает все буферы библиотеки, гарантируя, что строка "Hello!" не будет выведена дважды [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12).
    
- Переменная `local_var` демонстрирует механизм **Copy-on-Write**: её изменение в потомке не влияет на значение в родителе.
    
- `wait(&status)` — блокирующий системный вызов, который приостанавливает выполнение родителя до изменения состояния потомка (обычно до его завершения). Для ожидания конкретного потомка используется `waitpid()`.
    
- Макросы `WIFEXITED` и `WEXITSTATUS` используются для корректного извлечения кода возврата из составного значения `status` [](https://precise.sao.ru/hq/sts/linux/doc/kernel/kernel24/lki-2.html#ss2.12).

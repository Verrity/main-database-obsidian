Условные переменные — это примитив синхронизации, предназначенный для блокировки потока до наступления определенного условия, связанного с данными, защищенными мьютексом [](http://www.regatta.cmc.msu.ru/doc/usr/share/man/info/ru_RU/a_doc_lib/aixprggd/genprogc/condition_variables.htm#HDRD3A449E842MANU)[](https://studfile.net/preview/17174214/page:5/). Их ключевая особенность — атомарная операция освобождения мьютекса и перевода потока в состояние ожидания.

### 1. Фундаментальный принцип: Связка с мьютексом

Условная переменная никогда не используется сама по себе, а всегда в паре с мьютексом. Эта связка критически важна для предотвращения состояния гонки (race condition) между проверкой условия и переходом потока в режим ожидания [](https://www.daemon-systems.org/man/cv_wait_sig.9.html)[](https://studfile.net/preview/17174214/page:5/).

Рассмотрим классическую диаграмму, демонстрирующую взаимодействие потоков через условную переменную:

```mermaid
sequenceDiagram
    participant Поток_А as Поток А (Производитель)
    participant Поток_Б as Поток Б (Потребитель)
    participant Мьютекс as Мьютекс (mutex)
    participant CV as Условная переменная (cond)
    participant Данные as Общие данные (buffer)

    Поток_Б->>Мьютекс: pthread_mutex_lock(&mutex)
    activate Поток_Б
    Поток_Б->>Данные: Проверка условия (buffer == 0)
    Данные-->>Поток_Б: Условие истинно (буфер пуст)

    Поток_Б->>CV: pthread_cond_wait(&cond, &mutex)
    Note over Поток_Б,CV: Атомарная операция:
    CV-->>Мьютекс: 1. Освобождает mutex
    Мьютекс-->>Поток_Б: (разблокирован)
    CV->>Поток_Б: 2. Блокирует Поток Б
    deactivate Поток_Б

    Поток_А->>Мьютекс: pthread_mutex_lock(&mutex)
    activate Поток_А
    Поток_А->>Данные: Изменение данных (buffer = i)
    Поток_А->>CV: pthread_cond_signal(&cond)
    CV-->>Поток_Б: Пробуждение Потока Б
    Поток_А->>Мьютекс: pthread_mutex_unlock(&mutex)
    deactivate Поток_А

    Поток_Б->>Мьютекс: Попытка захвата mutex
    Note over Поток_Б,Мьютекс: Пробужденный поток Б пытается<br/>захватить mutex перед выходом из wait
    Поток_Б->>Мьютекс: pthread_mutex_lock(&mutex) (блокируется, если mutex занят)
    Мьютекс-->>Поток_Б: mutex захвачен
    activate Поток_Б
    Поток_Б->>Данные: Повторная проверка условия
    Поток_Б->>Мьютекс: pthread_mutex_unlock(&mutex)
    deactivate Поток_Б
```

Главная "магия" заключается в функции `pthread_cond_wait`. Как видно на диаграмме, поток, вызвавший эту функцию, должен заранее захватить мьютекс. Затем `pthread_cond_wait` выполняет две операции атомарно [](https://ru.stackoverflow.com/questions/1234642/%d0%9a%d0%b0%d0%ba-%d1%80%d0%b0%d0%b1%d0%be%d1%82%d0%b0%d0%b5%d1%82-%d1%8d%d1%82%d0%be%d1%82-%d0%bf%d1%80%d0%b8%d0%bc%d0%b5%d1%80/1483282#1483282)[](http://www.regatta.cmc.msu.ru/doc/usr/share/man/info/ru_RU/a_doc_lib/aixprggd/genprogc/condition_variables.htm#HDRD3A449E842MANU)[](https://www.daemon-systems.org/man/cv_wait_sig.9.html):

1. Освобождает захваченный мьютекс.
    
2. Переводит вызывающий поток в состояние ожидания на данной условной переменной.
    

Атомарность этих двух действий гарантирует, что никакой другой поток не сможет "вклиниться" и изменить состояние данных между моментом освобождения мьютекса и моментом, когда поток будет официально помечен как ожидающий. Если бы мьютекс освобождался до перехода в режим ожидания, существовало бы окно, в котором другой поток мог бы изменить условие и отправить сигнал, но первый поток еще не начал бы ожидать, что привело бы к потере сигнала и вечной блокировке [](https://manpages.opensuse.org/Leap-16.0/man-pages/futex.2.en.html)[](https://man.freebsd.org/cgi/man.cgi?query=futex&sektion=2&apropos=0&manpath=Debian+13.0+unstable).

### 2. Низкоуровневая реализация в Linux: Футексы (Futexes)

В современных ядрах Linux условные переменные (как и другие примитивы синхронизации из библиотеки `NPTL`) реализованы поверх механизма **футексов (fast userspace mutexes)** [](https://docs.linuxkernel.org.cn/locking/futex-requeue-pi.html)[](https://manpages.opensuse.org/Leap-16.0/man-pages/futex.2.en.html)[](https://man.freebsd.org/cgi/man.cgi?query=futex&sektion=2&apropos=0&manpath=Debian+13.0+unstable).

Футекс — это ядерный примитив, который позволяет блокировать поток, ожидая изменения значения 32-битной переменной в пользовательском пространстве. Логика работы `pthread_cond_wait` на основе футекса выглядит следующим образом:

1. **Проверка в Userspace:** Поток, владея мьютексом, проверяет предикат (например, `buffer == 0`). Если условие не выполнено, поток готовится к блокировке.
    
2. **Подготовка:** Поток атомарно изменяет состояние футекса, ассоциированного с условной переменной, сигнализируя о наличии ожидающего.
    
3. **Системный вызов:** Поток выполняет системный вызов `futex` с операцией `FUTEX_WAIT` (или её вариациями для PI). Важнейший нюанс: ядро перед блокировкой выполняет атомарную проверку: **значение футекса по адресу `uaddr` всё ещё равно ожидаемому?** [](https://manpages.opensuse.org/Leap-16.0/man-pages/futex.2.en.html)[](https://man.freebsd.org/cgi/man.cgi?query=futex&sektion=2&apropos=0&manpath=Debian+13.0+unstable). Если за время между шагом 2 и шагом 3 другой поток изменил значение (например, отправил сигнал), ядро не будет блокировать поток и вернет ошибку `EAGAIN`, заставляя поток перепроверить условие. Это и есть механизм предотвращения потерянных пробуждений.
    
4. **Блокировка:** Если проверка прошла, ядро переводит поток в состояние ожидания (`TASK_INTERRUPTIBLE` или `TASK_UNINTERRUPTIBLE`) и помещает его в очередь ожидания, связанную с данным футексом. Мьютекс к этому моменту уже освобожден (на шаге 2 в userspace, до входа в ядро, или как часть атомарной операции в ядре, в зависимости от конкретной реализации).
    

### 3. Ключевые операции

- **`pthread_cond_wait(cond, mutex)`**:
    
    - **Вход:** Поток должен быть владельцем `mutex`.
        
    - **Действие:** Атомарно освобождает `mutex` и блокирует поток до получения сигнала по `cond` [](http://www.regatta.cmc.msu.ru/doc/usr/share/man/info/ru_RU/a_doc_lib/aixprggd/genprogc/condition_variables.htm#HDRD3A449E842MANU).
        
    - **Выход:** После получения сигнала и перед возвратом из функции поток автоматически и атомарно **повторно захватывает `mutex`** [](https://manpages.ubuntu.com/manpages/trusty/man9/cv_timedwait.9freebsd.html)[](https://www.daemon-systems.org/man/cv_wait_sig.9.html). Это означает, что после возврата из `pthread_cond_wait` поток снова владеет мьютексом. Именно это свойство объясняет необходимость цикла `while` вокруг `pthread_cond_wait` (см. ниже).
        
- **`pthread_cond_signal(cond)`**:
    
    - **Действие:** Пробуждает как минимум один поток, ожидающий на данной условной переменной. Нет гарантии, какой именно поток будет разбужен (например, с высшим приоритетом — не гарантируется) [](https://manpages.opensuse.org/Leap-16.0/man-pages/futex.2.en.html). Пробужденный поток становится кандидатом на выполнение.
        
    - **Важное замечание:** Хотя в примерах часто показывают вызов `signal` до освобождения мьютекса, в реальности нет строгих требований к этому. Однако, держать мьютекс захваченным во время вызова `signal` может привести к лишним переключениям контекста: пробужденный поток тут же попытается захватить мьютекс, но обнаружит, что он занят, и снова уйдет в сон [](https://www.daemon-systems.org/man/cv_wait_sig.9.html).
        
- **`pthread_cond_broadcast(cond)`**:
    
    - **Действие:** Пробуждает **все** потоки, ожидающие на данной условной переменной [](https://manpages.ubuntu.com/manpages/trusty/man9/cv_timedwait.9freebsd.html)[](https://manpages.opensuse.org/Leap-16.0/man-pages/futex.2.en.html). Используется, когда изменение состояния делает возможным продолжение работы для нескольких потребителей одновременно.
        

### 4. Критически важный паттерн: "Spurious Wakeups" и цикл `while`

Код с использованием условных переменных всегда обрамляется в цикл `while`, а не в `if` [](https://ru.stackoverflow.com/questions/1234642/%d0%9a%d0%b0%d0%ba-%d1%80%d0%b0%d0%b1%d0%be%d1%82%d0%b0%d0%b5%d1%82-%d1%8d%d1%82%d0%be%d1%82-%d0%bf%d1%80%d0%b8%d0%bc%d0%b5%d1%80/1483282#1483282)[](https://www.daemon-systems.org/man/cv_wait_sig.9.html)[](https://studfile.net/preview/17174214/page:5/). Это связано с двумя факторами:

1. **Ложные пробуждения (Spurious Wakeups):** Реализация (как в ядре, так и в пользовательских библиотеках) допускает ситуацию, когда поток может выйти из состояния ожидания на условной переменной **без** получения соответствующего сигнала `signal` или `broadcast` [](https://www.daemon-systems.org/man/cv_wait_sig.9.html).
    
2. **Инвариантность условия:** Даже если пробуждение было вызвано сигналом, нет гарантии, что условие, которого ждал поток, всё ещё истинно. К моменту, когда поток получит процессорное время и захватит мьютекс, другой поток мог уже перехватить ресурс или изменить состояние.
    

Поэтому корректный паттерн использования всегда таков:

```c unfold
pthread_mutex_lock(&mutex);
while (проверка_условия_ложна) {
    pthread_cond_wait(&cond, &mutex);
}
// Здесь условие истинно, и мьютекс снова захвачен
/* ... выполнить действия ... */
pthread_mutex_unlock(&mutex);
```
### 5. Реализация в ядре: `struct cv` и `cv_wait`

В ядре Linux (и других UNIX-подобных ОС, например FreeBSD/NetBSD) используется похожий механизм, но с другими именами функций [](https://manpages.ubuntu.com/manpages/trusty/man9/cv_timedwait.9freebsd.html)[](https://www.daemon-systems.org/man/cv_wait_sig.9.html). Например, в ядре NetBSD:

- `cv_init()`, `cv_destroy()` для управления объектом условной переменной.
    
- `cv_wait(struct cv *cvp, kmutex_t *mtx)`: Работает идентично `pthread_cond_wait`. Переданный мьютекс `mtx` должен быть захвачен. Функция атомарно освобождает его и блокирует поток [](https://www.daemon-systems.org/man/cv_wait_sig.9.html).
    
- `cv_signal()` и `cv_broadcast()` для пробуждения.
    
- Существуют таймаутные версии: `cv_timedwait()`, `cv_wait_sig()` (ожидание, прерываемое сигналом) [](https://manpages.ubuntu.com/manpages/trusty/man9/cv_timedwait.9freebsd.html)[](https://www.daemon-systems.org/man/cv_wait_sig.9.html).
    

В ядре условные переменные используются для синхронизации доступа к ограниченным ресурсам (например, ожидание памяти) или завершения операций ввода-вывода [](https://www.daemon-systems.org/man/cv_wait_sig.9.html).

### Резюме

Условная переменная — это высокоуровневый механизм синхронизации, построенный над мьютексами. Её ключевая особенность — способность **атомарно освободить мьютекс и перевести поток в состояние ожидания**. В Linux реализация базируется на футексах, что позволяет минимизировать количество переходов в ядро. Корректное использование всегда подразумевает повторную проверку условия в цикле для защиты от ложных пробуждений и состояний гонки.

# Представим следующую ситуацию
Потоки изменяют переменную, есть поток наблюдатель, который следит когда переменная достигнет какого-то значения, и только тогда с ней что-то надо будет сделать.

#### Задача на mutex
Можно выполнить такую задачу, используя только мютексы, но это не всегда хорошо. Например, два потока инкрементируют переменную, и когда переменная достигнет нужного значения, мы хотим, чтобы третий поток изменил её значение другим образом.
**Такая задача на мютексах:**

```C
#include <stdio.h>
#include <pthread.h>

#define N 5

long a = 0;
pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;

void* thread_watch(void* args)
{
	while(1) {
		pthread_mutex_lock(&m1);
		if (a >= 50 000 000) {
			a += 2 000 000;
			pthread_mutex_unlock(&m1);
			pthread_exit(0);
			
		} else {
			pthread_mutex_unlock(&m1);
		}
	}
}

void* thread_calc(void* args)
{
	int i, tmp;
	for (i = 0; i < 90 000 000; i++){
		pthread_mutex_lock(&m1);
		a = a + 1;
		pthread_mutex_unlock(&m1);
		
		// tmp = a;
		// tmp++;
		// a = tmp;
	}
	return NULL;
}

int main(void)
{
	int i, i1;
	int* s;
	pthread_t thread[N];

	for (i = 0; i < N; i++){
		pthread_create(&thread[i], NULL, thread_calc, NULL);
	}

	for (i1 = 0; i1 < N; i1++){
		pthread_join(thread[i1], NULL, thread_calc, NULL);
	}
	
	printf("a = %ld\n", a);

	return 0;
}
```

Здесь мой поток будет постоянно захватывать мютекс и следить за значением переменной, в нужный момент изменяя её. Проблема в том, что поток наблюдатель будет постоянно мешать потокам, которые что-то считают, потому что постоянно будет перехватывать мютекс.


Правильно было бы проснуться только, когда переменная достигнет нужного значения.

-------------
#### Задача на Condition

```C
#include <stdio.h>
#include <pthread.h>

#define N 5

long a = 0;
pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;
pthread_cond_t cond1 = PTHREAD_COND_INITIALIZER; // Есть функция для инициализации condition, но требует свойства. Этот макрос создаёт condition с атрибутами по умолчанию.

void* thread_watch(void* args)
{
	pthread_mutex_lock(&m1);
	pthread_cond_wait(&cond1, &m1); // Это блокирующая функция, составляющая поток уснуть до тех пор, пока один из потоков не вызовет функцию pthread_cond_signal(). Мы захватили мютекс, но не освободили его, именно для этого мы передаём его в функцию. Внутри вызова мютекс освобождается, а мы блокируемся, пока не придёт сигнал. Как только придёт сигнал, мы снова захватим мютекс и войдём в свою критическую секцию.
	a += 2 000 000;
	pthread_mutex_unlock(&m1);
}

void* thread_calc(void* args)
{
	int i, tmp;
	for (i = 0; i < 90 000 000; i++){
		pthread_mutex_lock(&m1);
		if (a == 50 000 000) {
			pthread_cond_signal(&cond1); // Генерируем сигнал
			a++;
		} else {
			a++;
		}
		pthread_mutex_unlock(&m1);
	}
	return NULL;
}

int main(void)
{
	int i, i1;
	int* s;
	pthread_t thread[N];

	for (i = 0; i < N; i++){
		pthread_create(&thread[i], NULL, thread_calc, NULL);
	}

	for (i1 = 0; i1 < N; i1++){
		pthread_join(thread[i1], NULL, thread_calc, NULL);
	}
	
	printf("a = %ld\n", a);

	return 0;
}
```

**Как только вы сделали *pthread_mutex_unlock(&m1)* в *thread_calc*, другой поток-счётчик находился в состоянии ожидания *pthread_mutex_lock(&m1)* и он получил свой квант времени раньше, чем  *thread_watch*. Он мог захватить мютекс, инкрементировать и выйти раньше, чем поток *thread_watch*, так что когда *thread_watch* проверит значение переменной, оно уже может измениться. Так что если не стоит ожидать, что значение переменной будет ровно 50 000 000, так, если вы используете условные переменные, не стоит ожидать точно ожидаемого значения**

Такой подход более эффективен, но если то цикл в 1000 итераций, то большого прироста по сравнению с mutex ожидать не стоит.
